Appendices and reference sections follow at the end for advanced features and troubleshooting.

What WaxFrame Does
The concept in plain English

WaxFrame runs multiple AI assistants on your document at the same time. Most of them act as reviewers — they each read your document independently and produce numbered, specific suggestions for improvement. One AI acts as the Builder — it reads every suggestion from every reviewer, decides which ones are the strongest, and rewrites the document incorporating them. You review the result, resolve any disagreements between AIs, and run another round if needed. Each round the document improves. You stop when it is good enough.

WaxFrame works whether you are starting from a blank page, improving an existing draft, uploading a file, or pasting content in. It handles any kind of written document — business proposals, technical reports, cover letters, blog posts, marketing copy, executive summaries, policy documents, résumés, and more.

The three screens

WaxFrame has three main screens. You move through them in order when starting a new session.

ScreenWhat you do there
SetupConnect your AI assistants by saving their API keys. Choose which AI will act as the Builder. This is a one-time setup — your configuration is saved automatically and is there every time you return.
ProjectName your project, write your goal, and provide the document you want to work on — or choose to start from scratch and let the AIs write the first draft.
WorkRun rounds, read the updated document, resolve conflicts between AIs, refine further if needed, and export when you are done.
What you will need before you start

WaxFrame connects to AI providers using API keys. An API key is a unique code — like a password — that gives WaxFrame permission to use a specific AI on your behalf. You need at least two API keys from any combination of the supported providers: ChatGPT (OpenAI), Claude (Anthropic), Gemini (Google), Grok (xAI), Perplexity, or DeepSeek.

You pay each AI provider directly for the tokens you use. WaxFrame itself does not charge per use — it requires only a one-time license fee after your first three free rounds.

If you are not sure how to get API keys or which providers to use, open the API Key Guide ↗ — it has step-by-step instructions and direct links to each provider's signup and key console pages.

Free Trial & Licensing
Three free rounds — no credit card required

Every new installation of WaxFrame includes three free rounds. A round is one complete cycle — all your reviewer AIs process the document and the Builder produces an updated version. Three rounds is enough to see meaningful improvement on most documents. The free round count is stored in your browser and persists across sessions — it does not reset when you start a new project.

Purchasing a license

After your three free rounds, WaxFrame requires a license key to run additional rounds. Purchase one at weirdave.gumroad.com/l/WaxFrame. Your key arrives by email immediately after purchase. The license is a one-time purchase with no subscription or renewal fee.

How WaxFrame runs — browser-based, no install

WaxFrame is a browser-based application. There is nothing to install — you open index.html in any modern web browser and the app runs entirely in that browser window. Your API keys, settings, and session data are saved in your browser's local storage, not in the folder. This means the files in the folder never change as you use the app.

⚠️ Because your data lives in your browser, clearing your browser's site data or cache will permanently erase your saved API keys, settings, and any unsaved session. Use Menu → Backup Session regularly to save a copy of your session that you can restore later.

Setting up easy access

Once you have index.html open, bookmark it in your browser using Ctrl+D on Windows or Cmd+D on Mac so you can open it quickly in future. On Windows you can also right-click the file in File Explorer and choose Send to → Desktop (create shortcut). On Mac, drag the file to your Dock to pin it.

Step 1 — Set Up Your Worker Bees
What you are looking at

After clicking Let's get started → on the welcome screen, you arrive at the Setup screen. The screen is divided into two sides. The left side — headed Your Worker Bees — is what you are working with in Step 1. The right side — headed Choose Your Builder — is Step 2 and is covered in the next section. Do not worry about the right side yet.

Your Worker Bees are the AI assistants that act as reviewers each round. They each read your document independently and produce a numbered list of specific suggestions. You need at least two Worker Bees with saved API keys before WaxFrame will allow you to proceed. A status indicator at the bottom of the screen shows whether this requirement is met.

The toolbar — buttons across the top of the Worker Bees panel

Directly below the Your Worker Bees heading is a row of buttons. Here is what each one does, in the order they appear left to right:

API Key Guide Opens the API Key Guide — a separate page that walks you through getting an API key from each supported AI provider, with direct links to each provider's signup page and key console. Open this if you do not yet have keys or are not sure where to get them. This page opens in a new tab and does not close WaxFrame.
Add Custom AI Opens a form where you can connect an AI that is not in the default list — for example, Mistral, Together AI, Ollama running locally on your machine, or your company's internal AI platform. See Appendix A — Adding a Custom AI for full instructions. You do not need this for standard providers like ChatGPT, Claude, or Gemini — those are already in the list below.
Import from Model Server Opens a form that connects to a local or enterprise AI server (such as Open WebUI, Ollama, or LM Studio) and fetches the full list of available models, letting you add multiple AIs at once. This is an advanced feature for users running their own AI infrastructure. See Appendix B — Importing from a Model Server for full instructions.
Hide All Defaults Hides all six default AIs (ChatGPT, Claude, Gemini, Grok, Perplexity, DeepSeek) from the list at once. Use this if you are working in an environment where only internal or custom AIs are permitted and you want a clean list. Hidden AIs retain their saved keys and can be restored at any time — see Reset to Defaults below.
Open API Websites Opens the API console or billing dashboard for every AI currently in your hive simultaneously, each in its own new browser tab. Use this when you want to check your usage, billing, or key status across all your providers without navigating to each one manually. Your browser may block multiple tabs opening at once — if you see a pop-up blocker notification, click it and choose Allow, then click the button again.
Reset to Defaults Restores the six default AIs if you have hidden or removed any of them. Your saved API keys are preserved — this only affects which AIs appear in the list, not the keys themselves. Any custom AIs you have added are removed by this action. WaxFrame asks you to confirm before proceeding.
Test All Keys Sends a minimal test request to every AI that has a saved API key and displays the results in a panel that slides in from the bottom right of the screen. Each AI is tested one at a time in sequence. The panel shows a pass or fail result for each AI. Click ▶ Details on any row to expand the full test detail — including the exact endpoint used, the JSON body that was sent, the HTTP status code and response time, and the raw JSON response received. Use this before your first round to confirm all your keys are working.
The AI list — what each row shows

Below the toolbar is the list of your Worker Bees. By default, six AIs are shown: ChatGPT, Claude, DeepSeek, Gemini, Grok, and Perplexity. Each row in the list contains the following elements:

AI icon and name The logo and name of the AI provider. This is display only — it cannot be clicked or changed for default AIs.
↗️ (link icon) A small arrow icon next to the AI name. Clicking it opens that provider's API key console or signup page directly in a new tab — a shortcut to getting or managing the key for that specific AI without hunting for the URL yourself.
Hide Hides this specific AI from the list without deleting its key. Useful if you want to temporarily stop using one AI without removing its configuration. A banner appears at the top of the list showing how many AIs are hidden, with a ↺ Restore Hidden button to bring them back.
Key status indicator A small icon to the left of the key field. A white square (⬜) means no key is saved. A key icon (🔑) means a key is saved and active.
API Key field A password-obscured input field with the placeholder text Paste key — Enter to save… Click into this field, paste your API key from the provider, and press Enter. The field saves automatically on Enter or when you click away. Once saved, the key status indicator changes to the key icon and the Test and ✕ Key buttons appear.
👁️ (eye button) Toggles the API key field between hidden (dots) and visible (plain text) so you can verify the key you pasted. Click again to re-hide it.
✕ Key Appears only when a key is saved. Clicking it removes the saved key from that AI. The AI will no longer participate in rounds until a new key is saved. WaxFrame asks you to confirm before deleting the key.
Test Appears only when a key is saved. Clicking it sends a single minimal test message to that AI and opens a result modal showing the exact endpoint used, the JSON body sent, the HTTP status code and response time, and the complete raw JSON response received. A passing result shows a green ✅ status. A failing result shows a red ❌ with a plain-English explanation — for example, "Bad or missing API key" for a 401 error, or "Wrong endpoint URL" for a 404. Always use this button after saving a new key to confirm it works before running your first round.
Model selector Appears below the key row once a key is saved. Shows the model currently selected for that AI — for example, gpt-4.1 for ChatGPT or claude-sonnet-4-6 for Claude. Click the dropdown to see available models fetched directly from that provider. WaxFrame uses sensible defaults for each provider and you do not need to change this unless you have a specific reason to use a different model.
Entering your API keys — step by step
  1. 1Locate the AI you want to connect in the list — for example, ChatGPT.
  2. 2Click the API Key field next to that AI — the field with the placeholder text Paste key — Enter to save…
  3. 3Paste your API key using Ctrl+V on Windows or Cmd+V on Mac, then press Enter. The key status indicator changes to 🔑 to confirm the key is saved.
  4. 4Click the Test button that now appears next to the key field. Confirm the test passes before continuing.
  5. 5Repeat steps 1–4 for at least one more AI. You need a minimum of two AIs with saved, working keys before you can proceed.

Do not have API keys yet? Click the API Key Guide button in the toolbar above the list — it has direct links to each provider's signup page and walks you through getting a key for each one.

The status bar — knowing when you are ready

At the very bottom of the Setup screen, a status bar shows exactly what is still needed before you can continue. It reads: To continue you need: followed by two indicators:

  • ✗ At least 2 API keys saved — shows a red ✗ until two or more AIs have saved keys. Turns green ✓ when the requirement is met.
  • ✗ Builder selected — shows a red ✗ until you have selected a Builder on the right side of the screen (Step 2). Turns green ✓ when selected.

The Continue to Project Setup → button at the bottom right of the screen remains inactive until both indicators are green.

Step 2 — Choose Your Builder
What you are looking at

The right side of the Setup screen is headed Choose Your Builder. This is where you designate which AI is responsible for actually rewriting your document each round. While your Worker Bees (reviewers) each produce numbered suggestions, the Builder reads all of those suggestions and the full document and produces the updated version. You are still on the same Setup screen as Step 1 — just shift your attention to the right panel.

What makes the Builder different from reviewers

Every Worker Bee reads the document and gives suggestions. The Builder does the actual writing. Because the Builder must process the entire document plus all reviewer suggestions every single round, it consumes significantly more tokens than any individual reviewer. This has two practical implications:

  • The Builder must have a paid API subscription with sufficient token capacity. Free-tier API accounts have low limits and will cause rounds to fail mid-way through.
  • Choosing a capable, reliable AI as your Builder matters more than your choice of reviewers. The Builder's output is what you see in the working document after each round.
The Builder picker — what you see

The right panel displays a grid of cards, one for each AI that has a saved API key. AIs without a saved key do not appear here — you must save a key for an AI in Step 1 before it can be selected as Builder.

Each card shows the AI's name and icon. Click a card to select that AI as your Builder. The selected card is highlighted and marked with a crown icon (👑). Only one AI can be the Builder at a time. You can change your Builder at any time, including from the Work screen during an active session.

Claude, ChatGPT, Gemini, and DeepSeek all handle large documents reliably. DeepSeek in particular offers excellent output at a significantly lower cost than the US providers, making it an efficient choice as Builder. If a round fails with a structure error, switch your Builder to ChatGPT or Gemini — they tend to be the most consistent at following strict output format requirements. For a full breakdown of token costs across providers, see What Are Tokens? ↗

Continuing to the Project screen

Once the status bar at the bottom shows both ✓ At least 2 API keys saved and ✓ Builder selected in green, the Continue to Project Setup → button at the bottom right of the screen activates. Click it to move to the Project screen. All your keys and your Builder selection are saved automatically — they will be there every time you return to WaxFrame.

Step 3 — Define Your Project
What you are looking at

The Project screen has two sides. The left side — headed Your Project — is where you name your project and write the goal that guides every AI. The right side — headed Starting Document — is where you provide the document you want to work on, covered in Step 4. Work through the left side first, top to bottom.

Project Name field

The first field at the top of the left panel has the placeholder text Project name — e.g. Q3 Executive Summary. Type a name for your project here. This name is used in the export filename and appears in the round history. It has no effect on how the AIs work — it is purely for your own organisation. Examples: Q3 Board Report, Marketing Email — Product Launch, Cover Letter — Senior Engineer Role.

Version field

The smaller field next to the project name has the placeholder text Version — e.g. v1. Enter a version identifier here. This is also used in the export filename. Use whatever versioning makes sense to you — v1, draft1, 2.0, final. Like the project name, it does not affect how the AIs work.

Length Constraint (optional)

Below the name and version fields is a section headed Length Constraint (optional). This lets you set a maximum length for your finished document. Use it when your document must fit within a specific limit — for example, a 500-word blog post, a 200-character social media caption, or a one-page résumé.

The section contains two inputs:

  • A number field with the placeholder e.g. 200 — enter your maximum here.
  • A dropdown menu with three options: Characters, Words, or Pages — select the unit that matches your limit.

When a length constraint is set, WaxFrame includes it in every prompt sent to your AIs and instructs them to actively keep the document within the limit. Leave both fields blank if your document has no length requirement.

Project Goal field

Below the name and version fields is the most important part of the Project screen: the Project Goal field. It is a large text editor with line numbers on the left side and the placeholder text Project goal — what are you trying to create or achieve? The more specific and detailed you are, the better your results will be.

Every AI reads your goal before touching your document. It tells them what the document is, who it is for, what it should and should not include, and what outcome you need. The goal works identically whether you are uploading a file, pasting text, or starting from scratch. The difference is only in what the AIs have to work with — not in how the goal is applied.

A vague goal produces a vague document. A specific, detailed goal produces something you can actually use.

What to include in your goal
ElementWhat to write and why
Document typeState the format clearly — a business proposal, a technical comparison, a cover letter, an executive summary. Be specific about what kind of document this is.
AudienceWho will read this? Your IT team? A potential client? A hiring manager? A VP with no technical background? The audience shapes the tone, language, and depth every AI will use.
ScopeWhat should the document cover — and what should it explicitly not cover? AIs expand into adjacent topics unless you draw the boundary clearly. If you want wireless networking only and not switching or routing, say so.
Desired outcomeWhat do you want the reader to do or feel after reading? Make a decision? Approve a budget? Submit an application? Give the AIs a finish line.
Tone and styleFormal or conversational? Technical or plain-English? First person or third? Confident or cautious? The more specific you are here, the more consistent the output will be across rounds.
Writing your goal when you already have a document

If you are uploading or pasting an existing document, your goal still matters — but your focus shifts from what to create to what to improve. Tell the AIs what already exists, what works and should be preserved, and what specifically needs to get better.

Too vague "Make this proposal better."

Much better "This is a wireless network infrastructure upgrade proposal for a 950,000 sq ft warehouse. The audience is the IT Director and VP of Facilities — technical on the left, budget-focused on the right. The technical sections and vendor comparison table are solid and must not be changed. Focus all improvements on the executive summary — it is currently too long and too jargon-heavy for the VP — and on the cost justification section, which needs a clearer ROI narrative. Tone must be confident and direct, not salesy. Do not add any new sections."

Not sure what to write? The Document Playbooks ↗ has ready-made goals for dozens of common document types. Open it from the Menu or the link above.

Writing your goal when starting from scratch

When you have no existing document, the AIs generate the first draft entirely from your goal. Be as detailed as possible — include document type, audience, purpose, scope, tone, approximate length, structure, and any specific sections or headings you want included.

Too vague "Write a report about wireless networking."

Much better "Create a technical comparison of enterprise wireless networking platforms — HPE Aruba, Cisco Meraki, Juniper Mist, and Extreme Networks. Audience is our IT infrastructure team evaluating a campus-wide network refresh. Scope is strictly wireless: access points, controllers, cloud management, and licensing models. Explicitly exclude switching, routing, firewalls, and security products. Tone is technical and neutral — this is an internal evaluation document, not a sales pitch. Include a summary comparison table at the top, then a dedicated section for each vendor covering strengths, weaknesses, and pricing model. Close with a shortlist recommendation. Target length is approximately 2,000 words."

The character counter and the 300-character threshold

To the right of the Project Goal heading is a live character counter showing the current line count, word count, and character count of your goal. When your goal exceeds 300 characters, the character count turns amber as a signal that the goal will be trimmed in Refine rounds.

Here is why this matters: WaxFrame sends your goal differently depending on which round you are on.

Draft round (Round 1) Your full goal is sent to every AI regardless of length. This is the round where the AIs generate or first review the document and they need all the context you can give them.
Refine rounds (Round 2 onward) A trimmed version of your goal is sent, labelled Project Context. The trim targets approximately 300 characters but always cuts at a sentence boundary so nothing is left mid-thought. The document already exists — the token budget is better spent on the document content and reviewer suggestions than on repeating the full goal.

Put your single most important instruction in the first sentence of your goal. That sentence is guaranteed to be included in every Refine round regardless of goal length.

The Refine Preview button — see exactly what your AIs will receive

Next to the character counter is a button labelled ▸ Refine Preview. Clicking it opens a preview showing the exact text that will be sent to your AIs in Refine rounds, updated live as you type.

  • If your goal is under 300 characters, the preview shows your full goal and a note confirming it will be sent in full to every round.
  • If your goal is over 300 characters, the preview shows the trimmed version — cut to the nearest sentence boundary — so you can see exactly where the cutoff falls and adjust your opening sentence if needed.

The Refine Preview button is always available whenever your goal field contains text. On screens with limited height, the side-by-side preview panel collapses and the button is the only way to access the preview — it is always there.

The ⓘ button — About Your Project Goal

Next to the Your Project heading at the top of the left panel is a small ⓘ info button. Clicking it opens an overlay that summarises how the goal is used in Draft vs Refine rounds, explains the Refine Preview, and reminds you of the 300-character threshold. It is a quick reference to everything in this section of the manual.

Clear Goal and Clear Project buttons

At the bottom of the left panel are two buttons:

  • ✕ Clear Goal — erases only the text in the Project Goal field. Your project name, version, and all other settings are kept.
  • 🗑 Clear Project — erases everything on the Project screen: name, version, goal, length constraint, and export filename. Your API keys and Builder selection in Setup are not affected.

Step 4 — Provide Your Starting Document
What you are looking at

The right side of the Project screen is headed Starting Document, with a small ⓘ info button next to the heading. This is where you tell WaxFrame what to work with. There are three ways to provide a starting point, selected using the three buttons at the top of this panel.

Important: These three buttons — 📄 Upload File, 📋 Paste Text, and 💡 Start from Scratch — are mode selectors, not action buttons. Clicking one selects that mode and changes what appears below the buttons. You then interact with the area below them to actually provide your content. A brief instruction line directly below the buttons updates to tell you exactly what to do next.

Export Filename

Below the mode selector buttons and above the content area is a row labelled 💾 Export filename. This field controls what your downloaded document file will be named when you export it at the end of your session. It uses two tokens:

  • {name} — replaced with your project name, with spaces and special characters converted to underscores.
  • {version} — replaced with your version number.

The default value is {name}_{version}, which produces a filename like Q3_Executive_Summary_v1.txt. You can customise this — for example {name}_FINAL_{version} — or leave it as the default. A live preview of the resulting filename appears to the right of the field as you type, using your actual project name and version. Click the small ⓘ button next to the label for a full explanation of available tokens.

📄 Upload File — I have a document saved on my computer

Click 📄 Upload File to select Upload mode. A drop zone appears below — a dashed bordered area with the text Drag & drop or click to browse. To upload your file:

  • Click anywhere inside the drop zone to open your operating system's file browser. Navigate to your file and select it.
  • Or drag your file from any folder on your computer and drop it directly onto the drop zone.

Supported file formats: Word documents (.docx), PDFs (.pdf), PowerPoint presentations (.pptx), plain text files (.txt), and Markdown files (.md). WaxFrame extracts the text content from the file and loads it into the working document automatically. A confirmation message appears below the drop zone once the file is processed successfully, showing the filename and word count.

PDF tip: PDF extraction works best with standard digitally-created documents — files exported from Word, Google Docs, or similar applications. Scanned documents and PDFs created from images do not contain extractable text and will produce garbled or empty results. If your PDF was scanned, use the 📋 Paste Text option instead and copy the text from a version of the document you can select text in.

⚠️ Large file tip: If you import a very large file — a dense report, a long PowerPoint, or any document that produces a very high word count — and a round later fails with a structure error, this usually means the document exceeded your Builder's token limit for a single request. Switch your Builder to ChatGPT or Gemini using the Change Builder button on the Work screen and retry. Both handle large context windows reliably.

📋 Paste Text — I have text I want to copy in

Click 📋 Paste Text to select Paste mode. A text editor appears below the buttons — the same line-numbered editor used on the Work screen. Click anywhere inside the editor to place your cursor, then paste your text using Ctrl+V on Windows or Cmd+V on Mac. You can also type directly into the editor.

Use this option when your content comes from a website, an email, a Google Doc, another application, or any source where you can copy and paste text but do not have or do not want to use a file. There is no size limit on pasted text other than your Builder's token capacity.

The editor uses a fixed 80-character-wide column — the same width as the working document on the Work screen. This is intentional and ensures consistent formatting throughout your session.

💡 Start from Scratch — I do not have a document yet

Click 💡 Start from Scratch to select Scratch mode. A confirmation panel appears indicating that WaxFrame will generate the first draft entirely from your project goal. There is nothing else to do in this panel — no file to upload, no text to paste.

In this mode, your project goal on the left side of the screen is the only context the AIs have. Write it in as much detail as possible before launching. The more specific and complete your goal, the closer the first draft will be to what you actually need.

The ⓘ button — About Starting Document

The small ⓘ button next to the Starting Document heading opens an overlay that explains the three mode buttons and what to do after selecting each one — a quick in-app reference to this section of the manual.

Step 6 — The Work Screen
What you are looking at

The Work screen is where all rounds happen. It has a top bar, a three-column main area, and a footer bar at the bottom. Here is what each area contains.

The top bar

Running across the very top of the Work screen:

Menu button Three horizontal lines in the top left. Opens the slide-in navigation menu, which gives you access to the User Manual, API Key Guide, Document Playbooks, session backup and import, license entry, and navigation between screens.
Project name and version Shows your project name and version number. Clicking on this area opens a Project Goal viewer where you can read and edit your goal without leaving the Work screen.
Round badge In the centre of the top bar. Shows the current round number and phase — for example Round 1 — Draft or Round 3 — Refine Text. Updates automatically after each round completes.
📖 History Opens the Round History panel — a log of every completed round in the current session. Each entry shows the round number, timestamp, and a summary. Click any entry to view the full document as it was at the end of that round, and optionally restore it as the current working document if you want to roll back.
📝 Notes Opens the Notes drawer — a text area where you can write instructions specifically for the Builder for the next round only. Notes are cleared after each round. Use this to give targeted direction: "Tighten the third paragraph," or "The conclusion needs to be more decisive," or "Do not add any new bullet points."
🏁 Finish Opens the Finish panel where you can export your document, save a session snapshot, or start a new project. Covered in full in Step 10.
License badge A small indicator showing your current license status — either the number of free rounds remaining or a Pro indicator if you have a valid license.
The left column — The Hive

The left column is headed The Hive with a small ⓘ info button. This panel shows your AI reviewers and their status during each round.

Edit Hive button Opens a panel where you can toggle individual AIs on or off for the current session without going back to Setup. Step 7 — Run Your First Round
Clicking Smoke the Hive

Click the Smoke the Hive button in the footer bar to start a round. All your active reviewer AIs begin processing simultaneously — having more AIs does not make the round take longer, since they all run at the same time. You can watch each AI's progress in the Hive panel on the left — each icon animates as its AI works.

Do not close the browser tab or navigate away while a round is running. If the round is interrupted, the document will not be updated and the round will not be saved to history, though your API call tokens will still have been consumed.

What happens during a round
  1. 1WaxFrame sends each reviewer AI a prompt containing your goal, the current document, and instructions for their role. All reviewer AIs receive the same prompt simultaneously.
  2. 2Each reviewer AI returns a numbered list of specific suggestions. You can see these in the Live Console as they arrive, and in full detail in the right column after the round.
  3. 3Once all reviewers complete, WaxFrame sends the Builder AI a prompt containing the current document, all reviewer suggestions, and your goal. The Builder incorporates the strongest suggestions and produces an updated document.
  4. 4The updated document appears in the Working Document panel. Any significant disagreements between your AIs are flagged as conflicts in the right column for your decision. The round is saved to history automatically.
Draft round vs Refine rounds

The round badge in the top bar shows the current phase.

Round 1 — Draft Your first round is always a Draft round. Your full project goal is sent to every AI. If you chose Start from Scratch, the AIs produce a first draft. If you provided a document, the AIs review it and the Builder produces an improved version. After Round 1, the phase changes to Refine automatically.
Round 2+ — Refine Text Every subsequent round is a Refine round. The AIs read the current document and give targeted improvement suggestions — tightening language, fixing inconsistencies, strengthening arguments, improving structure. Your goal is sent as a trimmed Project Context (approximately the first 300 characters, cut at a sentence boundary).

Step 8 — Review Results and Resolve Conflicts
Reading the updated document

After a round completes, the centre column shows the updated document. Read it carefully before running another round. Each version is saved automatically to the Round History — click 📖 History in the top bar to access any previous version and restore it if needed.

What is a conflict?

A conflict appears in the right column when your AIs gave the Builder genuinely incompatible suggestions — recommendations that point in different directions strongly enough that the Builder flagged the decision for you rather than choosing arbitrarily. Common conflict scenarios: one AI suggested removing a section while another suggested expanding it; AIs disagreed on tone, phrasing, or structure for a key passage; reviewers proposed different conclusions.

Conflicts are not a sign something went wrong. They are a signal that your AIs found a real decision point in the document — one that genuinely depends on your intent and cannot be resolved by the AI alone.

Resolving conflicts

Each conflict in the right column shows:

  • The original text that was in dispute
  • Two or more options proposed by your AIs, each clearly labelled
  • A Custom option where you can type your own resolution
  • A Keep Original option to leave that section exactly as it is

Read each option and click the one that best matches your intent. Once you have made a selection for every conflict, click Apply Decisions to update the document. Resolved conflicts are locked — WaxFrame informs all AIs about your decisions in future rounds and instructs them not to re-raise the same issues.

Using conflicts to steer the document

Rather than treating conflicts as a problem to dismiss quickly, use them as a tool. They often surface real questions about the document's direction — questions about emphasis, tone, structure, or argument that you will need to answer to get the document where you want it. A conflict is your AIs showing you where they genuinely disagree, and your resolution tells the entire hive which direction to take in future rounds.

Step 9 — Run More Rounds
The rhythm of a session

After your first round, the process repeats: click Smoke the Hive, wait for the round to complete, read the updated document, resolve any conflicts, make any direct edits you want, and decide whether to run another round. Most documents reach a high quality in three to five rounds. Some need more; some need fewer. The right time to stop is when the document reads the way you want it to.

Using the Notes drawer for targeted direction

Click 📝 Notes in the top bar to open the Notes drawer. This is a plain-text area where you write direct instructions for the Builder for the next round only. Notes are not sent to reviewers — only the Builder receives them. They are cleared automatically after each round so they do not carry over unintentionally.

Use Notes when you want something specific to happen in the next round that your goal alone would not produce:

  • "Tighten the executive summary — it should be no more than three sentences."
  • "The third paragraph in the Cost section is redundant — remove it and tighten the section."
  • "The conclusion is weak. Rewrite it to be more decisive and action-oriented."
  • "Do not change the Technical Specifications section under any circumstances."

The Notes drawer has a 📋 Copy button to copy your notes and a ✕ Clear button to erase them before a round if you change your mind. Close the drawer with the ✕ Close button when you are done.

Editing the document directly

The Working Document editor is fully editable at any time between rounds. If you can see exactly what needs to change — remove a paragraph, fix a factual error, rewrite a heading — just do it directly. The AIs will work with whatever is in the editor when the next round starts. If you remove something intentionally and want to make sure the AIs do not reintroduce it, add a Note: "I have removed [X] — do not reintroduce it."

Adjusting your hive between rounds

Click Edit Hive in the left column to toggle individual AIs on or off for the current session. Click Change Builder to switch which AI is doing the writing. Neither action requires going back to Setup or losing your current session. Both take effect on the next round you run.

Using Send to Builder instead of a full round

When you have made direct edits to the document and want the Builder to integrate and clean them up — without running a full reviewer cycle — click Send to Builder in the footer bar instead of Smoke the Hive. The Builder rewrites the document based on your current version and your Notes (if any), but the reviewers do not run. This saves time and tokens when you already know what direction to take and just need the Builder to execute it cleanly.

Knowing when you are done

Watch for these signals that the document is ready:

  • Your AIs are agreeing more than they are disagreeing — conflicts are rare or minor.
  • Conflicts are about word choices rather than structural or argument issues.
  • You are accepting most suggestions without hesitation.
  • Reading the document feels like reading a finished piece, not a draft.

Running more rounds past this point produces diminishing returns. When the document reads the way you want it to, stop and export.

Appendix A — Adding a Custom AI
When to use this

The six default AIs (ChatGPT, Claude, Gemini, Grok, Perplexity, DeepSeek) cover the most widely-used public AI providers. Use Add Custom AI when you want to connect an AI that is not in that default list — for example, Mistral, Together AI, Cohere, Ollama running locally on your machine, LM Studio, or any other AI service that exposes an API endpoint accepting chat completion requests in OpenAI-compatible, Anthropic, or Google format.

To open the form: go to the Setup screen and click Add Custom AI in the toolbar above the Worker Bees list.

The Add Custom AI form — every field explained
Quick Add (optional) A dropdown with presets for Mistral, Together AI, Cohere, Ollama (local), and LM Studio (local). Selecting a preset auto-fills the endpoint URL and API format fields. You can then fill in the remaining fields manually. If your provider is not listed, leave this on the default and fill all fields manually.
URL (required) The full API endpoint address WaxFrame will send requests to, including the path. For most OpenAI-compatible providers this ends in /v1/chat/completions. Example: https://api.mistral.ai/v1/chat/completions. Get this URL from your provider's API documentation.
Name (auto-detected) The display name for this AI in WaxFrame — what you will see in the list and in the Hive panel. Auto-filled by Quick Add and also attempted from the URL you enter. You can change it to anything.
API Format (required) The API format this provider uses. Three options: OpenAI compatible (for most third-party providers — Mistral, Together AI, Ollama, LM Studio, and most others), Anthropic (for direct Claude API endpoints), or Google (for direct Gemini API endpoints). When using Quick Add this is filled automatically. If unsure, ask your provider: "Is this API OpenAI-compatible?" The answer is almost always yes.
Model (required) The exact model ID string the provider expects in API requests — for example mistral-small-latest or llama-3.1-8b-instruct. This must be exact — even a single typo causes every API call to fail with a model-not-found error. Use the Fetch Models button (see below) instead of typing this manually whenever possible.
API Key (optional) Your API key or bearer token for this provider, if authentication is required. Paste it in. For local models like Ollama with no authentication, leave this blank. There is an 👁️ button to toggle visibility of the key field.
Fetch Models button

After entering your URL and API key, click Fetch Models. WaxFrame sends a request to the provider's models endpoint and retrieves the list of available models. A dropdown appears below the Model field showing all available models — click the one you want to use. This is significantly more reliable than typing a model ID manually and eliminates the risk of typo errors.

Test Connection — required before adding

Click Test Connection before adding the AI. WaxFrame sends a minimal test message to the endpoint and opens a result panel showing the exact endpoint used, the full JSON body sent, the HTTP status code and response time, and the complete raw JSON response received. A passing test shows a green success message. A failing test shows a red error with a plain-English explanation:

  • 401 / 403 — bad or missing API key
  • 404 — wrong endpoint URL or path
  • 405 — method not allowed — the endpoint may not support chat completions at this path
  • 429 — rate limited — wait a moment and try again
  • Network Error — the endpoint is unreachable, or a CORS policy is blocking the request from a local file origin

⚠️ The Add to Hive button only appears after a successful test result. If you change any field after a passing test, the result resets and you must test again before the Add to Hive button reappears. This prevents accidentally adding a misconfigured AI.

Appendix B — Importing from a Model Server
When to use this

If you have a local or enterprise AI server running — such as Open WebUI, Ollama, or LM Studio — and it hosts multiple models, Import from Model Server lets you fetch the complete list of models from that server and add several AIs to your hive at once, instead of adding them one by one using the Add Custom AI form. To open the form: go to the Setup screen and click Import from Model Server in the toolbar.

The Import from Model Server form — every field explained
Quick Add (optional) A dropdown with presets for Open WebUI, Ollama (local), and LM Studio (local). Selecting a preset auto-fills the Chat Endpoint and Models Endpoint fields with the standard paths for that server type. You then confirm or adjust the base URL portion to match your actual server address.
Chat Endpoint (required) The full URL WaxFrame uses to send chat completion requests to this server — the URL that ends in the chat completions path. For Open WebUI this is typically https://your-server.com/api/chat/completions. For Ollama it is http://localhost:11434/v1/chat/completions.
Models Endpoint (required) The full URL WaxFrame uses to fetch the list of available models from your server. For Open WebUI this is typically https://your-server.com/api/models. For Ollama it is http://localhost:11434/api/tags. This endpoint is queried when you click Fetch Models.
API Key (optional) Your API key or bearer token if your server requires authentication. Leave blank for local servers with no authentication configured.
Fetching and selecting models
  1. 1Fill in the Chat Endpoint, Models Endpoint, and API Key fields (or use Quick Add to pre-fill them).
  2. 2Click Fetch Models. WaxFrame queries your Models Endpoint and retrieves the full list of available models. A status message and the raw response from your server appear below the button so you can confirm the connection worked.
  3. 3Click Select & Add to Hive →. A checklist panel opens showing every model returned by your server.
  4. 4Check the models you want to add to your hive. Use Select All or Select None to manage the list quickly.
  5. 5Click Add Selected to Hive. WaxFrame adds each selected model as a separate AI in your Worker Bees list, using the Chat Endpoint you provided and the model ID for each one.

If Fetch Models returns a CORS or Network Error, your server may be blocking requests from local file origins. Ask your IT team to configure the server to allow file:// origin requests, or serve WaxFrame from a local web server so requests originate from an HTTP address.

Many organisations run their own AI infrastructure — platforms like Open WebUI connected to internally hosted models, or enterprise subscriptions to OpenAI or Anthropic with private endpoints and access controls. Setting up an internal-only hive

If your organisation uses only internal AI models and you do not want the six default public AIs (ChatGPT, Claude, etc.) visible in your list, hide them using the Hide All Defaults button in the toolbar on the Setup screen. This hides all six at once. You can hide them individually using the Hide button on each AI row instead. Hidden AIs retain their saved keys and can be restored at any time using Reset to Defaults or the ↺ Restore Hidden button that appears in a banner at the top of the list.

CORS and running from a local web server

WaxFrame opens as a local HTML file in your browser. Some internal AI servers block requests from file:// origins as a CORS policy — this is a security measure that prevents arbitrary web pages from connecting to internal servers. If you see a Network Error or CORS error when testing a connection, you have two options:

  • Ask your IT team to add file:// to the allowed origins on the AI server's CORS configuration.
  • Serve WaxFrame from a simple local web server. On most machines this can be done by opening a terminal in the WaxFrame folder and running python -m http.server 8080, then opening http://localhost:8080 in your browser instead of the file directly. Requests from localhost are typically allowed by internal servers.

When Things Go Wrong
The document is going in the wrong direction
SituationWhat to do
Tone is wrong throughoutOpen Notes before the next round and write explicitly: "The tone throughout the entire document must be [formal / conversational / technical / plain-English]. Rewrite any passages that do not match this." Be specific about what you want.
AIs keep adding content you do not wantEdit your project goal to explicitly exclude it. Add a Note reinforcing the exclusion. If one specific AI keeps doing it, use Edit Hive to remove that AI for a round or two.
The document has drifted badly from where you want itClick 📖 History and restore an earlier version. Add a Note explaining the direction the next round should take. If your goal was too vague, edit it on the Project screen before running another round — click the project name in the top bar to open the goal editor without leaving the Work screen.
It would be faster to start overClick 🏁 Finish → Start New Project. This clears the document and history but keeps your API keys. Return to the Project screen, rewrite your goal with tighter scope, and begin again.
A round failed with a structure error

This means the Builder returned output that WaxFrame could not parse — usually because the document was too long or complex for that AI to follow the required output format precisely under that token load. Click Change Builder in the Hive panel, switch to ChatGPT or Gemini, and run the round again. Your document was not changed by the failed round, and the failed attempt is recorded in your Round History for reference.

An AI keeps re-raising a conflict you already resolved

Open Notes before the next round and write: "[AI name] is repeatedly re-raising [issue]. This decision is final — [chosen text] is locked and must not be changed. Do not allow any variation of this suggestion." If the problem persists after two rounds of this instruction, use Edit Hive to remove that AI from the session.

An API key is not working

Click the Test button next to that AI on the Setup screen. The result panel shows the exact error. Common causes: the key was copied incorrectly (extra space at the start or end is a frequent culprit), the key has been revoked or expired at the provider's end, the account has run out of credit, or the account is on a free plan that does not include API access. Visit that provider's API console directly — click the ↗️ icon next to the AI in the list to open it — and verify your key and account status.

The Smoke the Hive button is not active

WaxFrame requires at least two AIs with saved, working keys and a Builder selected before the Smoke the Hive button activates. If it is greyed out or inactive, go back to the Setup screen (via the Menu) and verify that at least two AI key fields show the 🔑 icon and that a Builder is selected on the right side of the Setup screen.

A provider link or page has moved

AI provider websites change their navigation regularly. If a link in WaxFrame leads to a dead or moved page, open the API Key Guide ↗ from the Menu — it is updated with each WaxFrame release and contains current direct links for every supported provider.

API Costs & Billing
You pay the AI providers directly — not WaxFrame

WaxFrame does not charge per use. The cost of running rounds comes from the API calls WaxFrame makes to your AI providers on your behalf — and those are billed directly to your account at each provider. WaxFrame requires only a one-time license fee after the first three free rounds.

API costs are measured in tokens — small units of text (roughly 0.75 words per token). You are charged for both the tokens sent in the prompt and the tokens returned in the response. For a detailed explanation of how tokens work and how to estimate your costs, see What Are Tokens? ↗

DeepSeek consistently produces high-quality output at a fraction of the cost of OpenAI, Anthropic, or Google. It is an excellent choice as both a reviewer and Builder when managing costs matters.

Monitoring your spending

Click Open API Websites on the Setup screen to open the billing or usage dashboard for every AI in your hive simultaneously. Check your usage after your first few sessions to understand your actual per-round cost. Provider pricing changes periodically — bookmark your billing dashboards and check them at least monthly.

Perplexity — enable auto-billing before your first use

Perplexity requires you to explicitly enable auto-billing in your account settings before API calls will succeed. If Perplexity's Test button returns a billing-related error, log into your Perplexity account at perplexity.ai/settings/api, enable auto-billing, and retry the test.

Pop-up blocker and opening multiple billing pages

The Open API Websites button opens multiple browser tabs simultaneously. Most browsers block this by default and show a pop-up blocked notification in the address bar. Click the notification and choose Always allow pop-ups from this site, then click Open API Websites again. This only needs to be done once per browser.
