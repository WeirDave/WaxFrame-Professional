WaxFrame Professional - Tools Folder
=====================================

This folder contains development tooling for regenerating documentation
screenshots. None of it ships to end users; it's only used when updating
the visuals in the user manual and README.


What's in here
--------------

capture-screenshots.ps1
    Entry point. A thin PowerShell launcher that finds Chrome and Node on
    your machine, then runs capture.mjs. This is the file you actually run.

capture.mjs
    The driver. A Node ES module that does all the real work:
      - Spawns a tiny static HTTP server for the repo on port 8731
      - Launches a fresh headless Chrome with remote debugging enabled
      - Connects via the Chrome DevTools Protocol over WebSocket
      - Seeds a sanitized fake hive (and a Cookies session for the work
        shot) into the headless browser's localStorage and IndexedDB
      - Navigates to each screen, waits for a readiness signal that proves
        content has rendered, then captures the screenshot
      - Cleans up the temp Chrome profile when done

README.txt
    This file.


How to run it
-------------

From a PowerShell prompt in this folder:

    pwsh -ExecutionPolicy Bypass -File ".\capture-screenshots.ps1"

That produces 16 PNGs in the repo's screenshots\ folder:
  - screenshot_welcome_dark.png + light
  - screenshot_setup1_dark.png + light  (Worker Bees)
  - screenshot_setup2_dark.png + light  (Builder)
  - screenshot_setup3_dark.png + light  (Your Project)
  - screenshot_setup4_dark.png + light  (Reference Material)
  - screenshot_setup5_dark.png + light  (Starting Document)
  - screenshot_settings_dark.png + light
  - screenshot_work_dark.png + light  (populated Cookies session)

Two screenshots are NOT regenerated by the tool because they need live
round states the tool can't currently produce:
  - screenshot_building_dark.png
  - screenshot_convergence_dark.png
These are captured by hand from a real session when the live-round UI
changes meaningfully.


Flags
-----

-Theme dark|light|both    Default is both. Use one to halve runtime.
-OnlyWork                 Re-capture just the work screen. Useful when
                          iterating on the seeded session payload.

Examples:
    pwsh -ExecutionPolicy Bypass -File ".\capture-screenshots.ps1" -Theme dark
    pwsh -ExecutionPolicy Bypass -File ".\capture-screenshots.ps1" -OnlyWork


Requirements
------------

Chrome (64-bit recommended)
    Either Chrome or Edge works, but Chrome is preferred. The launcher
    checks Program Files first, then Local AppData. If neither has a
    usable browser, it exits with a clear error.

    Why 64-bit: the Edge x86 stub at Program Files (x86) hands headless
    launches off to whatever browser the user already has open, which
    breaks isolation. The launcher prefers a real 64-bit binary.

Node.js 22 or newer
    The driver uses built-in WebSocket and fetch, which Node 22 added.
    Earlier Node versions will fail with a clear version-check error.

    Install from nodejs.org. No npm packages needed; capture.mjs uses
    only Node built-ins.


How the seed data works
-----------------------

The work screen and the Builder picker only render meaningful content
when the app has a configured hive (AIs + keys). To produce honest
documentation screenshots without requiring the tool's user to log into
real accounts, capture.mjs embeds:

  HIVE_SEED_BASE   - The 6 default AIs with placeholder keys
                     (every key is literally the string "sk-CAPTURE-DEMO")
  PROJECT_SEED     - The Chocolate Chip Cookies project metadata
  SESSION_SEED     - The post-Round-1 work-screen state (document,
                     resolved conflicts, builder responses, live console)

All real API keys and all masked-key log fragments (like "key: <prefix>****<suffix>")
were scrubbed before embedding. The seeded values are fake. The seeded
session content (the recipe text and AI responses) is real output from
the canonical Cookies dry-run, kept for fidelity.

These seeds are injected into a brand-new isolated Chrome profile created
in your system's temp folder, then the entire profile is deleted when the
tool finishes. Your real Chrome browser, your real localStorage, and your
real keys are never touched.


Architecture notes
------------------

The capture driver waits for a per-screen readiness handshake before
taking each screenshot:
  - Builder shot waits until builderPickGrid has at least 2 children
  - Bees shot waits until aiSetupGrid has at least 2 children
  - Work shot waits until screen-work is visible after the IDB session
    has been seeded
  - Other screens just wait until the target screen element is visible

This eliminates the timing race the older version had (which sometimes
required 8-10 retries to land 16 clean shots). One clean pass should now
produce all 16 correctly.

No capture logic lives in the app code. The app is shipped clean and
unmodified by any of this tooling.


Troubleshooting
---------------

"Node.js not found"
    Install Node 22+ from nodejs.org. Tool checks PATH and the standard
    install locations.

"No Chrome or Edge found"
    Install Chrome from google.com/chrome. The tool needs a 64-bit
    Chromium-based browser binary.

A specific shot reports "FAILED (not-ready: ...)"
    The readiness check timed out. The reason in parentheses tells you
    what failed - e.g. "builder-cards=0" means the seeded hive didn't
    populate, "screen-hidden" means navigation didn't land on the right
    screen. Re-run once; if it persists, something in the app changed
    that the readiness check needs to be updated for.

PNG file size around 86 KB
    That's a roughly-empty page. Most real screens come out 300-500 KB.
    An 86 KB shot usually means the seed didn't take or the screen
    didn't render content - check the console output for the readiness
    reason.
