Observations

- 2026-06-06 — gh CLI + PAT-authed workflow is the answer to programmatic release publication from Claude Code on web. Install gh in the sandbox via direct binary download from github.com/cli/cli/releases (apt-get is available but the direct tarball is simpler), authenticate with `gh auth login --with-token` via a fine-grained PAT scoped to Contents:write on the target repo. `gh release create vX.Y.Z --target <FULL-SHA> --title "WaxFrame vX.Y.Z Pro" --notes-file ...` does tag creation + release publication atomically. CRITICAL gotcha: --target MUST be the full 40-char SHA, not the short hash; the API returns "target_commitish is invalid" with a confusingly-worded 422 if you pass the short SHA. PAT lives in /root/.config/gh/hosts.yml in the sandbox only; revoke at github.com/settings/tokens after the session ends since it appeared in chat history.

- 2026-06-06 — Web UI "Create new tag on publish" defaults to current HEAD, NOT to the commit each release version was authored against. When David published v3.63.190 through .195 via web UI in succession with HEAD at bd0b721 (v3.63.195's commit), all 6 tags ended up pointing at bd0b721. Caught by `gh api repos/.../git/refs/tags/vX.Y.Z --jq .object.sha`; fixed by `gh api ... --method PATCH -f sha=<correct-full-sha> -F force=true` on each tag. Lesson: if publishing past releases via the web UI, set "Target" to the specific commit, not the branch tip. If using gh release create, --target with the full SHA handles this correctly.

- 2026-06-06 — Screenshot regeneration was FLAGGED for ~50 releases because the failure was silent. The capture script exited 0 even with 2 of 16 shots failing because the underlying capture.mjs prints "FAILED" but doesn't exit non-zero on individual shot failures. Considered hardening that — making the script exit non-zero if ANY shot fails — but the override decision was no: sometimes a single shot's readiness check needs updating, and you'd rather have 14 fresh + 2 stale than 0 fresh. The real fix is review the per-shot OK/FAILED summary every time you run it. Add a release-check or smoke-test that diffs screenshot mtime vs the index.html mtime, maybe — but that's a future enhancement.

- 2026-06-06 — METHODOLOGY for honest engaged-time estimates on multitasking runs (David's proposal during the Joe Schmoe's analysis). Project-clock-reported "engaged" is over-stated whenever the user forgot to pause during idle gaps. Honest formula: sum the per-round AI processing time (parallel workers → max(worker_elapsed) + builder_elapsed since builders run after the worker batch completes; builder-only rounds are just builder elapsed) directly from the scout bundle ringBuffer entries. Then add a reasonable human-review estimate: ~2 min per review round (read the new doc + handle conflicts panel), ~30 sec per builder-only round (basically just hit Continue), plus a setup pad of ~5 min for first round. Result is a defensible lower bound for actual engagement that doesn't pretend phone-game / center-console / Claude-conversation time was active document review. Joe Schmoe's worked example: 11.2 min AI + ~18 min human + 5 min setup = ~30 min estimated true engaged, vs. 76 min reported, vs. 102 min wall-clock. The engaged-time enhancement (Backlog v10 Section 2) that auto-pauses on idle would close this gap but isn't shipped yet; until it does, USE THIS METHODOLOGY for any HISTORICAL RUNS entry where the user reports forgot-to-pause inflation.
