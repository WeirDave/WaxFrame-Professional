1. Lock button one-way — does not unlock
   ─────────────────────────────────────────────────────────
   WHAT:   In the Conflicts panel each applied change has a
           per-row lock button. Clicking it once locks the
           change (prevents Builder from overwriting it on
           later rounds). Clicking it again is supposed to
           unlock — but currently does nothing visible.
   WHY:    Users get stuck with a permanently locked change
           and have to clear the whole conflicts panel to
           recover. Removes a piece of fine-grained user
           control during refinement.
   STATE:  DEFERRED — waiting on user repro with the
           v3.63.165 diagnostic console line captured.
   WHERE:  lockAppliedChange()       at app.js:18485
           diagnostic console.log    at app.js:18504
                                     ('[wf:lock]', ...)
           silent return on empty    at app.js:18514
           renderConflicts target    document.getElementById(
                                     'conflictsPanel')
   DONE:   Clicking a locked lock button toggles it back to
           unlocked state in both the data model AND the
           rendered UI; subsequent Builder rounds can edit
           that line again.
   HYPOTHESES (ranked by likelihood):
     a. change.locked is false at click time — rendered
        state and data state have diverged, most likely if
        the history array got reassigned between renders.
     b. The UNLOCK branch enters correctly but the silent
        return at app.js:18514 fires because change.new
        became an empty string.
     c. The UNLOCK branch runs cleanly but
        getElementById('conflictsPanel') returned null
        inside renderConflicts() and the function silently
        bailed without re-rendering.

OPEN FEATURES
════════════════════════════════════════════════════════════

1. Builder-incapable family list expansion
   ─────────────────────────────────────────────────────────
   WHAT:   Some AI model families can't reliably act as
           Builder — they break the %%CONFLICTS_START%%
           marker block, truncate output mid-document, or
           hallucinate formatting that breaks the parser.
           WaxFrame maintains a filter list of these
           families. Today the list contains Jamba (added
           v3.63.132) and a handful of others.
   WHY:    Every time a new provider/family is added (e.g.
           Cohere Command-A-Plus, Together AI families), it
           needs to be evaluated as Builder. Some pass, some
           fail. The list has to expand as the provider
           catalog grows.
   STATE:  OPEN — ongoing housekeeping. No specific candidate
           queued for this entry.
   WHERE:  Builder-capable filter logic in app.js
           (recommendation engine + Builder dropdown
           filtering). Cross-reference: Jamba filter at the
           recommendModel() path.
   DONE:   A new family proves structurally incompatible as
           Builder in two consecutive measured runs → it
           goes on the list; recommendation engine excludes
           it from Builder picks; Builder dropdown either
           hides it or shows it disabled with a tooltip.


2. "+ Add variant" UX on default AI cards
   ─────────────────────────────────────────────────────────
   WHAT:   Each default AI card (Claude, ChatGPT, Gemini,
           Grok, Mistral, Perplexity) gets a "+ Add variant"
           button. Clicking it spawns a sibling slot under
           that parent: same provider, same saved API key,
           but its own model picker. So a user can run
           Claude Sonnet + Claude Opus + Claude Haiku in
           the SAME hive as three independent reviewers, all
           billed through the same Anthropic key.
   WHY:    The concurrency override feature shipped in
           v3.63.299 only earns its keep when a hive has
           2+ AIs sharing one provider. For default-AI users
           that scenario is currently impossible without
           re-keying through the "Add Custom AI" flow with
           the matching preset — which adds an entry under
           CUSTOM AIs rather than as a sibling of the default
           card, reading as a hack rather than a first-class
           feature. Surfacing variants as siblings of the
           default card makes "I want multiple Claudes" feel
           native, and unlocks the concurrency override UI
           for the default-AI population. Server imports
           (Open WebUI etc.) already have natural multi-AI-
           per-provider behavior; this feature gives default
           providers the same.
   STATE:  OPEN — design direction understood; not started.
           Concurrency limiter + override now correctly route
           through the BASE provider (v3.63.301), so adding
           variants Just Works for throttling once the UI +
           storage shape land.
   WHERE:  AI setup grid render:
             renderAISetupGrid + _buildAISetupRow in app.js.
           Variant storage: most likely a new key
             hive.aiVariants = { [parentId]: [{id, model, ...}] }
           or extend customAIConfigs with a parentId field.
           Variant resolution at round time:
             activeAIs already iterated by callAPI — variants
             would surface as additional ai objects with
             ai.parentId set so the key is read from
             API_CONFIGS[parentId]._key (single key, many AIs).
   DONE:   Each default AI card has a "+ Add variant" button
           that spawns a sibling row sharing the parent's
           key but with its own model picker; variants
           persist across reloads; removing a variant doesn't
           orphan it from the hive; removing the parent
           prompts to either remove all variants too or
           promote one variant to parent; concurrency
           override Settings still shows one row per BASE
           provider (already true in v3.63.301); recommend-
           model + tier classifier work per-variant; the
           Builder badge applies to the variant the user
           crowned, not the parent.


3. Pause/Resume at reviewer seam
   ─────────────────────────────────────────────────────────
   WHAT:   A refinement round has natural phases:
             (1) reviewers analyze the working document,
             (2) conflicts surface in the Conflicts panel,
             (3) user decides on any USER DECISION conflicts,
             (4) Builder applies the decisions and writes
                 the new working document,
             (5) the next round starts.
           The "reviewer seam" is the moment between phases
           (1) and (4) — reviewers done, Builder not yet
           fired. This feature would expose a Pause button
           at that seam: freeze the session, persist state,
           and let the user resume later.
   WHY:    Long sessions (15+ rounds) overlap with meal
           breaks, meetings, rate-limit windows. Today the
           user either powers through or loses the round.
           A clean pause point preserves engaged-time
           accuracy (smart clock pauses) and gives the user
           breathing room without dropping work.
   STATE:  OPEN — not started.
   WHERE:  Round state machine in app.js. Existing precedent:
           smart project clock idle-pause (v3.63.197) shows
           how session state can be paused cleanly.
   DONE:   At the reviewer seam, a Pause button appears;
           clicking it saves full round state to
           localStorage, halts the Builder dispatch, and
           shows a Resume button; clicking Resume re-fires
           the Builder with the captured reviewer outputs;
           smart clock is paused for the duration; no data
           loss if the browser tab is closed (state survives
           a reload via localStorage).


4. File System Access API exploration
   ─────────────────────────────────────────────────────────
   WHAT:   The File System Access API is a modern browser
           API (Chrome, Edge, Opera; not Firefox or Safari)
           that lets a web app read and write files on the
           user's disk directly, with permission, instead of
           using <input type="file"> upload dialogs and
           download links. The user grants the app a handle
           to a folder or file once, and the app can then
           read/write that location continuously.
   WHY:    WaxFrame's Project, Template, and Checkpoint
           save/load flows today are upload-and-download.
           If we adopted FSA for browsers that support it,
           users could pick a working folder once and have
           Projects auto-save as they go, with checkpoints
           landing in the same folder — much closer to a
           "real" desktop app feel without leaving the
           browser. Firefox/Safari users would keep the
           current flow.
   STATE:  OPEN — research phase. No code yet.
   WHERE:  N/A — exploration. Likely landing zone is the
           project save/load code path in app.js + any
           settings UI for "Working Folder."
   DONE:   A spike branch demonstrates: (a) detect FSA
           support, (b) prompt for a working folder, (c)
           auto-save the current Project to that folder
           every N seconds, (d) load existing Projects
           from that folder on launch; UX comparison
           written up; go/no-go decision on shipping.


5. Prompt instruction modularization
   ─────────────────────────────────────────────────────────
   WHAT:   The instructions WaxFrame sends to each AI are
           still largely assembled inline in app.js. Reviewer
           instructions, Builder instructions, project goal
           fields, reference material, current document,
           resolved decisions, warnings, notes, and required
           response formats all flow through the same prompt
           assembly paths. This feature would break the
           role-specific instruction blocks out into explicit
           prompt modules so Reviewer and Builder instructions
           can be inspected, edited, tested, and versioned
           separately from the per-round data envelope.
   WHY:    Prompt changes are now core product changes. When
           instructions are embedded inside long assembly
           functions, it is too easy to accidentally nest
           instructions inside examples, blur Reviewer vs
           Builder roles, or change the data envelope while
           trying to tune behavior. Separate instruction
           modules reduce prompt regression risk and make the
           Prompt Editor path more realistic later.
   STATE:  OPEN — design direction understood; not started.
   WHERE:  Reviewer prompt assembly:
             buildPromptForAI(ai, []) in app.js
           Builder instruction source:
             BUILDER_INSTRUCTIONS in app.js
           Builder send paths:
             runBuilderOnly() and applyDecisions() in app.js
           Full-round fan-out:
             runRound() reviewer phase in app.js
   DONE:   Reviewer and Builder instructions live in separate
           named prompt modules/objects; the project/reference/
           document/history envelope is assembled independently;
           generated prompts are byte-for-byte equivalent where
           no intentional wording change is made; Deep Dive still
           captures the final prompt sent to each AI; a future UI
           can point at these modules without parsing app.js.

NOTES ON FORMAT
════════════════════════════════════════════════════════════
Every entry above uses the same five-field structure:
WHAT, WHY, STATE, WHERE, DONE. Going forward, any new
backlog entry MUST follow that structure so the document
stays pickup-ready for any reader.

State vocabulary:
  OPEN      — known and ready to work on; no blockers.
  PARTIAL   — work has started, more remains; current
              state described in the entry.
  PARKED    — design is understood but not on a near-term
              release path; revisit later.
  DEFERRED  — paused waiting on an external input
              (user repro, upstream decision, etc.).

When an item ships, it moves to CHANGELOG.md and is
DELETED from this file. No "CLOSED" section here — the
changelog is the system of record for completed work.
