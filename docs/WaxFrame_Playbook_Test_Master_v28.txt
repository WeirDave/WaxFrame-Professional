[BACKLOG-001] Change Builder modal — two-stage pick (vendor → model)
──────────────────────────────────────────────────────────
PRIORITY: medium-high (real UX gap, blocks cost optimization)
SCOPE: v3.53 candidate — meaningful enough to deserve its own release

PROBLEM
The current Change Builder modal commits the Builder identity on a single
click of the AI card. When the new Builder takes over, it inherits whatever
model was previously set for that vendor — which is typically the
REVIEWER-recommended model from the "Recommend Models for All" flow, not
the Builder-recommended one. Result: switching Builder to Gemini (for
example) silently sticks with the Reviewer-class Gemini model instead of
the Builder-class one that the recommendation engine already picked.

GOOD NEWS
The infrastructure is already built. recommendModelsForAll at app.js L3171
stores per-role caches:
  • waxframe_recommend_default-${provider}-reviewer
  • waxframe_recommend_default-${provider}-builder
And getBuilderRecommendation(aiId) at app.js L92 reads the Builder cache.
The data is just not being consumed by setBuilderFromModal at L871.

DESIGN (locked with David 2026-05-17)
Two-stage modal flow:
  1. Vendor grid (existing) — selects rather than commits on click
  2. New model dropdown appears below grid — single dropdown reused for
     whichever vendor was selected
  3. Dropdown pre-selects the Builder-recommended model from
     getBuilderRecommendation(id); falls back to current model if no
     Builder rec exists for that vendor
  4. New "Confirm" button commits both vendor and chosen model
  5. Cancel preserves prior state (vendor unchanged, model unchanged)

IMPLEMENTATION EDGES
  (a) Models source — waxframe_models_${id} localStorage cache from
      prior "Recommend Models for All" / "Refresh Models". If empty for
      a vendor, three options:
        (a-i)  Show only current model as the lone dropdown option
        (a-ii) Auto-trigger fetch with spinner when vendor is selected
        (a-iii) Inline "Refresh models for this vendor" link in dropdown area
      Recommend (a-iii) — explicit user action, no surprise async.
  (b) Power-user double-click escape hatch — should double-click on a
      vendor card commit immediately with the Builder-recommended model,
      bypassing the dropdown step? Open question for build time.
  (c) Dropdown state isolation — the dropdown selection must not leak
      into the actual model setting until Confirm fires. Cancel discards.

ESTIMATED EFFORT
~80–150 lines: new modal HTML, dropdown population logic, two-stage commit
handler, styling within existing modal aesthetic.

──────────────────────────────────────────────────────────
[WATCH-001] Word-boundary truncation in char-mode templates
──────────────────────────────────────────────────────────
SCENARIO
The Trim to Google Maps template targets 750–1,200 chars. In theory, an
AI could end its output exactly at the char ceiling and truncate
mid-word.

CURRENT POSITION
Not pre-engineering. LLMs generate token-by-token, and tokens align to
word boundaries (or at worst sub-word syllable boundaries) — they
essentially never truncate a word mid-character. The AI manages the
char target as a soft cliff at sentence/phrase breaks, sometimes
overshooting by a few chars rather than cutting cleanly.

TRIGGER FOR ACTION
If mid-word truncation is observed in any live Google Maps run, the
one-line fix is adding to BUILDER_INSTRUCTIONS or the template's
refMaterial directive:
  "End at sentence or word boundaries — never mid-word."

──────────────────────────────────────────────────────────
Template-name header on output document
──────────────────────────────────────────────────────────
PROPOSED: David proposed having the trim templates emit a header line
on output identifying the platform ("Trim to TripAdvisor" as line 1,
blank line, then review body). Intent: help users identify which file
is for which platform when comparing multiple versions in Downloads.

DROPPED BECAUSE: Implementation required either (a) length-guard would
fire ~20 chars over the cap on tight Google Maps runs, or (b) JS-side
length-counting logic would need to strip the header before measuring.
David's call: "if you think this is going to be a pain in the butt
because of those titles then just leave the stupid thing off — it was
just a suggestion and sometimes I make suggestions that don't work out
logistically."

──────────────────────────────────────────────────────────
Multiplier-based Source Size Check thresholds (1.5×, 1.2×)
──────────────────────────────────────────────────────────
PROPOSED: I (Claude) twice proposed multiplier-based thresholds for
the SSC oversize trigger (initially 1.5×, then 1.2× after Yelp+Manly
missed). David rejected: "I don't understand why you're doing math on
overages and stuff... if my hard high limit on Google is 800 characters
and I paste a document that's 1000 then it's fucking over." Final v3.52.5
design: source > max fires, no multiplier. Locked.

PROPOSED: Kai's three escalation tiers for Gemini cost warnings:
