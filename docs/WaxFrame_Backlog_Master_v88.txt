Observations

- v3.63.19 validate-on-save scope: catches auth-class failures (HTTP 401 / 403) at the moment of save on the Worker Bees screen so bogus keys surface ❌ before the user walks to the work screen. Per-provider routing covers all six default providers correctly: ChatGPT/Grok/Perplexity/DeepSeek use origin-based derive (handles Perplexity's endpoint shape that has no /v1/ segment), Claude routes through the CF Worker proxy (avoids browser CORS), Gemini uses the Google API with x-goog-api-key, Mistral hits api.mistral.ai/v1/models with Bearer. Initial implementation (build -018) used fetchModelsFromEndpoint and only worked for 4 of 6 providers — caught during same-session dogfooding when David X-Key'd Perplexity, typed a bogus key, and got no ❌ feedback. The rewrite mirrors fetchModelsForProvider's per-provider dispatch (api.js ~376) rather than relying on the generic derive path. Lesson: when adding a feature that needs auth-error signals, mirror the existing provider-aware code rather than the generic helper that swallows or mis-derives.

- v3.63.19 green Recommend button: third button-state UI convention added to WaxFrame (after Test green ✅ and the slow-responder indicator). Pattern is now: blue idle -> "Asking…" disabled in-flight -> green confirm with check-mark + 5-second timed revert. Race-protected by class-presence check in the setTimeout callback — if user clicks the action again during the green window, the new in-flight state takes priority and the revert becomes a no-op. Worth codifying as a button-state-transition rule for the next utility action button that ships.

- v3.63.18 dead-migration cleanup: removed three one-shot data migrations (V33210 recommend-cache reshuffle, V3605 + V3607 Together model-list filter/URL changes) plus a v3.30.2 one-time upgrade toast inside ensureOriginalModelBaseline. All four had been firing on every page load for users who had long since completed them, doing nothing each time. Their localStorage sentinels were swept in the same release via a small unconditional removeItem block at the top of DOMContentLoaded (removeItem is idempotent so no new sentinel needed — the sweep itself can run forever without becoming the next generation of cruft). Pattern lesson: one-shot migrations need a sunset, or they become permanent boot-time overhead and visible scar tissue in user localStorage. Going forward: when a migration is written, include a "remove after vX.Y.Z" comment in the doc-comment so future-you has a clear delete signal. Aggressive removal is appropriate when the project hasn't reached broad adoption — any returning user who somehow missed every release for a year can self-heal via cache wipe.

- v3.63.18 clearUploadedFile bug: the function had been silently leaking PDF state since the PDF re-extract feature was introduced. The fallback path in discardVerifyExtractedText carried a comment claiming "clearUploadedFile() handles ... source-type localStorage cleanup" — comment was lying; only filename was cleared. Code-comments-can-lie principle confirmed again. Audit reminder: when a function's doc-comment claims behavior, verify the body actually does it before trusting the comment.

- Layered code surfaces bugs that only appear under specific edge interactions. The clearUploadedFile leak and the v3.63.17 disabled-attribute trap both required a user to back-navigate after partial setup to hit them. Forward-only happy-path testing never sees either. Pattern: after every multi-step setup flow change, regression-test every Back-button path.

- v3.63.18 re-extract banner styling: pre-v3.63.18 the banner relied on .honeycomb-header class semantics that were gated to .work-panel-header — it never received the panel styling. Trying to add the class to the banner div was a non-fix because the CSS selector requires the parent class too. Right fix was to give .reextract-banner its own self-contained panel styling matching the visual weight of .honeycomb-header without inheriting from it.

- v3.63.17 Worker Bees paste-key bug = dead code from the pre-v3.31.0 single-list era (Internet + Server AIs combined; Server mode hid Internet defaults via isActive=false). v3.31.0 two-tab split made the gate unnecessary but the disabled attribute + matching CSS rule survived as orphan code. Bug only fired when continueFromBees stripped activeAIs to keyed-only, then user back-navigated — invisible under normal forward-only setup. Lesson: when a UX architecture changes (single-list -> two-tab), audit every gate that depended on the old shape, not just the ones that visibly fire under the new shape.

- Incognito is the cleanest "factory state" diagnostic available — no extensions, no LS_HIVE residue, no service worker. When a bug reproduces in main Chrome AND Edge but not Incognito, the differentiator is persisted state from prior visits (LS_HIVE in this case), not browser / extension / environment.

- P0 security cleared in one surgical release (v3.63.7). Both fixes followed the codebase's established two-layer trust-boundary pattern (import sanitization + render-sink belt for XSS; the redaction function + bundle-assembly redaction for content). safeRefId mirrors safeUrl exactly.

- safeRefId design: STRIP to [A-Za-z0-9_-] rather than regenerate, so legit ids pass through unchanged and the inline-handler lookups (which match by doc.id ===) stay intact. Only empty/all-bad ids fall back to a fresh generated id. Behaviorally tested: legit preserved, injection payloads stripped of all quotes/parens/brackets, empty falls back.

- The diagnostic redaction was leaking 3 ways despite the checkbox: LS_PROJECT raw (biggest — full document), plus ringBuffer + lastFailure inside the session that the redactor never visited. Lesson: when a redaction function exists, audit EVERY field that ends up in the export, not just the obvious docText/history — side-channels (debug buffers, separate localStorage blobs) are where content hides. consoleHTML included for completeness.

- Render-sink belt mutates doc.id in place when it's unsafe, persisting the normalized id so the rendered DOM id and the array id stay in sync (handlers look up by doc.id). Compute-once (single safeRefId call) avoids the regenerate-returns-different-value-each-call trap.

- v3.63.21 validation lesson: /v1/models is unreliable as a key-validation signal across providers. Perplexity's /v1/models endpoint is a PUBLIC CATALOG that returns 200 OK regardless of authentication, so bogus keys passed validation. We don't know which other providers have the same trait and can't audit every provider's documentation forever. The reliable validation is the chat-completion endpoint — testApiKey has always used it because chat-completion endpoints require real auth on every provider (that's what makes them billable). v3.63.21 leverages the existing API_CONFIGS shape (cfg.endpoint + cfg.headersFn + cfg.bodyFn) that testApiKey already uses, sending a minimal one-word probe instead of inventing a new validation path. Lesson: when there's a proven validation path in the codebase already (testApiKey), don't invent a parallel one — extend or share the path that's been working. Three releases (v3.63.19, v3.63.20, v3.63.21) to land on this approach when the answer was sitting in the codebase the whole time.
