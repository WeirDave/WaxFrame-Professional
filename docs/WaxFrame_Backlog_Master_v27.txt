CHANGES IN v22 (from v21, 2026-05-10):
  - v3.37.0 SHIPPED — Dual-path templates. Every template
    in the gallery now declares which starting condition it
    supports: "scratch" (user has no draft; hive creates one)
    or "refine" (user has a draft; hive polishes it). 17 of
    19 templates support both paths. Two single-path:
    Quick Start (scratch only — the onboarding demo) and
    Multi-Platform Review Rewrite (refine only — operates
    on a source review).
    Modal flow: open gallery → pick "Starting from scratch"
    or "Refining an existing draft" → category grid filters
    to templates supporting that path → click a template →
    Project Goal fields populate from the per-path content,
    and (scratch only, by default) a Reference Material card
    is injected with a questionnaire scaffold. Refine paths
    inject no RM card; instead the hint banner points the
    user to paste their existing draft into Starting Document.
    schema (templates.js):
      Top-level path-agnostic identity fields:
        id, name, icon, category, description, paths,
        pathContent
      pathContent[path] holds per-path fields:
        goalDocType, goalAudience, goalOutcome, goalScope,
        goalTone, goalNotes, refMaterial, hint, lengthMode,
        lengthLimit, lengthMin, lengthUnit
    Retired field: suggestedNotes. Content that previously
    lived in suggestedNotes is now routed to Reference
    Material as a card on apply (scratch path), tagged
    source: 'template' + templateOriginId. Subsequent
    applyTemplate calls sweep prior template-sourced cards
    so the RM panel stays clean and idempotent (switching
    templates cleans up after the previous one).
    routing (applyTemplate):
      Signature changed: applyTemplate(templateId, path).
      Resolves with fallback (warn + default to first
      supported path) if caller passes invalid path. Reads
      all per-path content from tpl.pathContent[path].
      RM sweep filters d.source !== 'template'; manually
      added cards (source: 'paste' or 'file') are preserved.
      Hint banner title now includes path suffix
      ("starting from scratch" / "refining an existing
      draft") so the user can see at a glance which path
      was applied.

[ ] ☐ Engaged-time tracking for runs  (NEW v9 — DOWNGRADED v17:
        the consistency bug was the Builder-Only project clock
        parity issue, fixed in v3.36.32. The broader feature
        below — auto-idle detection, manual pause UX — is now
        optional enhancement, not a bug fix)
    NEW from session 2026-05-10. Wall-clock session time is
    misleading because it includes idle periods when the user
    stepped away (lunch break, dinner, distracted by other
    work). The Publix potato salad recipe run shows "47 min"
    wall-clock; actual engaged time was a fraction of that.

    PROBLEM IT CAUSES: HISTORICAL RUNS times aren't comparable
    across runs. Two 4-round runs that both show "30 min" might
    actually have very different engaged times. This degrades
    the value of the time-component of measurement data.

    DESIGN OPTIONS:
      Option A (manual pause): add a Pause button next to the
      Auto/Manual control. User clicks before stepping away,
      clicks again on return. Simple, reliable, requires user
      discipline. The existing Auto-pause-on-Notes-open guard
      from v3.36.17 is a similar pattern — extend it.

      Option B (auto-detect idle): track last user-interaction
      timestamp (keystroke, click, scroll). If idle > N minutes
      (configurable, e.g., 5 min default), pause the round
      timer. Resume on next interaction. Smart but heuristic.

      Option C (both): manual pause for known breaks, auto-
      detect for forgotten ones. Best UX but most code.

    WHAT IT TOUCHES:
      • Round timer (currently wall-clock-based)
      • Console log timestamps (also wall-clock — these are
        good as-is for forensic purposes; engaged-time is a
        separate metric)
      • Transcript export "Session duration" line — should
        report both wall-clock AND engaged time for clarity
      • HISTORICAL RUNS rows in test master — going forward
        should record both metrics
      • Round badge / topbar elapsed display — show engaged
        time, not wall-clock

    NOT a blocker for current work — runs already in the wild
    have wall-clock data, that's what we have, we move on.
    Going forward, capturing engaged time would make the data
    more meaningful.

    PRIORITY: high but not P1 — affects measurement quality,
    not blocking any release. Lift when there's a quiet
    session with no other surgical work to do.
