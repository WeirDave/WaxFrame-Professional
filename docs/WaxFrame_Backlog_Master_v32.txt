  - SHIPPED v3.53.1 — Security audit follow-up part 2:
    consoleHTML sanitize on restore (Section 6.1.D Option A,
    David's preferred). Focused micro-release on top of
    v3.53.0 same evening — closes the XSS vector that
    existed on session restore from any backup file
    (including v3.53.0 backups, since the format didn't
    change). Two restore sites in js/storage.js (loadSession
    main IDB path + localStorage fallback path) now run
    stored consoleHTML through a new sanitizeConsoleHTML
    helper before innerHTML assignment.

    APPROACH (Option A per backlog v30 6.1.D):
      1. DOMParser parses raw HTML into a DETACHED
         HTMLDocument. <script>, <img onerror>, etc. do NOT
         fire during detached-document parsing.
      2. Walk the parsed tree and rebuild in the live
         document via createElement + textContent matching
         a strict per-tag schema mirroring consoleLog's
         output.
      3. Anything outside the schema dropped silently.

    SCHEMA: Top-level requires <div class="console-entry
    [console-{info|warn|error|success}]">. Children limited
    to:
      • <span> — class console-time or no class; text only
      • <a> — class console-link; href must be http(s)://;
        target=_blank, rel=noopener canonical-set; all other
        attrs stripped including any onclick
      • <button class="console-err-arrow"> — title length-
        bounded; onclick must match strict regex
        /^openConsoleErrorDetail\('cle_\d+_[a-z0-9]+'\)$/
        otherwise stripped (button renders inert)

    Any other tag dropped. Closed-list trade-off favoring
    security over feature drift — future consoleLog
    additions must update the sanitizer or restored
    sessions silently strip new content.

    BACKWARD COMPATIBLE: no backup format change. All
    existing backups restore through the sanitizer
    unchanged — legitimate consoleHTML passes through
    because the schema mirrors what consoleLog actually
    produces. Pre-existing limitation preserved:
    legitimate <a class="console-link"> elements lose
    their click handler on restore (the handler was a
    JS-side property, not a serialized attribute) and
    rely on the href fallback. Already true before this
    release.

[ ] Word-boundary truncation in char-mode trim templates
    --------------------------------------------------
    ADDED: 2026-05-17 (v28, post-Yelp template live
    testing).
    SCENARIO: The Trim to Google Maps template targets
    750–1,200 chars. In theory, an AI could end its
    output exactly at the char ceiling and truncate
    mid-word.
    CURRENT POSITION: Not pre-engineering. LLMs generate
    token-by-token, and tokens align to word boundaries
    (or at worst sub-word syllable boundaries) — they
    essentially never truncate mid-character. AIs manage
    the char target as a soft cliff at sentence/phrase
    breaks, sometimes overshooting by a few chars rather
    than cutting cleanly.
    TRIGGER FOR ACTION: If mid-word truncation is
    observed in any live Google Maps run, the one-line
    fix is adding to the Google Maps template's
    refMaterial directive: "End at sentence or word
    boundaries — never mid-word."
    HOW TO CHECK: review live Google Maps trim runs.
    If output ever shows a word getting cut, take action.

[ ] 6.2.D  Content Security Policy — long-term hardening
    Codex finding #7: no CSP declared. The app uses inline
    onclick handlers starting from the first screen, which
    would make a strict CSP harder to roll out but
    correspondingly more valuable.

    WHY VALUABLE: with API keys in localStorage, a CSP that
    blocks inline scripts and restricts script sources is
    a meaningful blast-radius reducer if any XSS does land.

    WHY HARD: rolling out a strict CSP requires migrating
    every inline onclick="…" handler to addEventListener
    bindings. Easily 200+ handler sites across index.html
    and the helper pages.

    APPROACH (when prioritized): incremental migration.
    Pick one screen, migrate its onclicks to JS-side
    delegation, ship. Repeat. Once all onclicks are gone,
    add the CSP meta tag and verify nothing breaks.

    EFFORT: large. v3.6+ candidate, not v3.52.x.

[—] 6.2.E  Client-side license enforcement — not actionable
    Codex finding #8: isLicensed() trusts a localStorage
    flag and the dev password hash is shipped client-side.

    NOT A FIX TARGET: this is the architectural reality of
    being a static-site app with no backend. Server-side
    license validation would require a backend, which would
    change the entire deployment model. License enforcement
    is a business control and a courtesy gate, not a
    security boundary. The Gumroad license-verify endpoint
    we DO call (api.gumroad.com/v2/licenses/verify) is the
    real validation point; clients that bypass it locally
    are bypassing the courtesy gate, not a real authorization
    boundary.

    KEEP IN MIND, NOT ACTIONABLE.

[ ] 6.2.F  Wider XSS sweep — other unsafe interpolations
    NEW IN v32 (2026-05-19). v3.53.2 closed the
    resolveAiIcon chain but the wider codebase still has
    other places where ai.name and ai.id get interpolated
    raw into HTML templates without escaping.

    KNOWN SITES (audit needed for completeness):
      • js/app.js Change Builder modal grid (~L767-773):
        title="${esc(ai.name)}" is escaped, but
        onclick="setBuilderFromModal('${ai.id}')" and the
        bare ${ai.name} in <span class="builder-pick-name">
        are NOT.
      • Likely others in the changeBuilderReason flow,
        hive setup, and ai-setup screens.

    THREAT: same shape as 6.2.B — a malicious imported
    AI config with a crafted ai.id (containing ') or
    ai.name (containing < or ") could escape attribute
    or JS-literal context and execute arbitrary code.
    Lower realistic likelihood than 6.2.B was — the
    icon path was rendered everywhere, these others
    are rendered in narrower flows — but the same
    underlying threat model.

    APPROACH: full-file grep for ${ai.name} and
    ${ai.id} in template strings; for each site decide

    EFFORT: medium. Probably 10-20 sites, ~100-200

Closes the last meaningful XSS sink from the 2026-05-17 Codex security audit — js/app.js resolveAiIcon previously interpolated user-controlled ai.name, ai.icon, and cssClass raw into HTML strings including inside an inline onerror= JS string literal, which would have let a malicious imported AI config (ai.name = `'"><img src=x onerror=alert(1)>`) execute attacker code on every render with full localStorage access (API keys, license key). New handler reads name/class/size from data-* attributes (which are HTML-escaped at write time and round-trip cleanly through DOM dataset API) then passes them to makeAiAvatar's DOM-API construction path (createElement + textContent + style.cssText). XSS-sink data path broken at every step. makeAiAvatarHTML got matching escapeHtml(cssClass) + Number(size) treatment. makeAiAvatar (DOM-Element version, already used createElement + textContent) got Number(size) coercion as defense in depth.
