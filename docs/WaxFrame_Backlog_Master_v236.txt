Observations

- 2026-06-06 — v3.63.183 through v3.63.189 published as a single rolled-up GitHub release rather than 7 separate ceremonies. Pattern decision: when intermediate ceremonies share a coherent theme (here: Project Templates polish + bee-strip iteration + the headline Custom Categories feature), publish as one release with a multi-section body. Versions in between remain unsplit in the CHANGELOG for archaeology, but the Releases sidebar stays clean. Tag pushes from the sandbox 403'd; tags were created via the web UI publish flow.

- 2026-06-06 — Cache-bust drift validator (release-check.mjs check 4) added in v3.63.190 work. Catches the v3.63.180-182 failure mode the project lived through three months ago — multiple "ceremonial" releases shipping CSS/JS against the same ?v= key. Sample output on the current state: "16 CSS/JS file(s) changed since v3.63.182; APP_VERSION advanced 3.63.182 → 3.63.189". The failure case ("CSS/JS shipped but APP_VERSION did NOT advance") is what the check is for; the success case logs the deltas so future archaeology is faster.

- 2026-06-06 — Worker Bees rework #24 ("drop sub-1366px responsive code") REMOVED from backlog. v3.63.166 density pass at style.css:2742 explicitly handles the 1366×768 Notes-button-truncated-to-'tes' failure mode by collapsing top-bar buttons to icon-only with tooltip labels under 1500px. Removing that code would re-introduce the bug the v3.63.166 pass shipped to fix.

- 2026-06-06 — v3.63.183 was net cleanup. Stripped the v3.63.180-182 Project Templates honeycomb experiment back to the existing theme surface ramp. Three iterations earlier in the session (honeycomb-in-header → honeycomb-as-modal-body → flat var(--surface) surfaces) walked back to the simplest theme that matched the rest of the app. Lesson: when the theme stack already has a working surface ramp, prefer it over inventing modal-specific texture treatments.

- 2026-06-06 — The honeycomb body background on the page (html/body, .hp-section-header, work-screen panels) stays. Only the Templates modal lost it. Different surface, different needs: the page can carry the texture because section headers float over it as glass strips; a modal can't because the modal overlay covers the body so there's nothing to show through. Keep the texture where the structural pattern supports it.

- 2026-06-06 — The cache-bust query string ?v=3.63.182 DRIFTED for the entire v3.63.180-182 cycle. Each release shipped CSS/JS changes against the same key. End result: CDN and browser caches kept serving the previous file even after deploy; David spent at least one session reverting work he thought was broken when the actual issue was stale cache. v3.63.183 bumped the key as part of stamping the work. Future point releases need to bump it every time CSS/JS lands, not just on the next "ceremonial" release. Worth adding to the release-check validator if not already there: if any of style.css / js/*.js changed since the last tagged release, the ?v= query in HTML must have changed too.

- 2026-06-06 — Andy is the first paid user (purchased 2026-06-05 at work, RFI use case). cleaner.html removed in the same release because the bug it repaired only ever affected David's own Dropbox-legacy custom-AI ID collisions; Andy has no such state.

- 2026-06-05 — Template Gallery theme polish needs both theme passes every time. v3.63.182 started as a dark-leaning patch, and David caught that light mode had not received the same care. Future visual work should verify light, dark, and auto-light/auto-dark before ceremony.

- 2026-06-05 — The amber rail system should stay structurally identical across themes. For the template/path cards the shared rule is `border-left: 4px solid ...`; theme overrides should change the color and surrounding surface, not the width.

- 2026-06-05 — The viewport-vs-screen-resolution confusion is the #1 source of "but my screen is 1920x1080 so why am I blocked" support friction. Surfacing the actual CSS-pixel viewport in the overlay disambiguates the diagnosis at zero cost — no external dependency, no link rot, no second-page-load. Should have been there from day one of the min-screen-overlay; flagged here as a lesson for any future "what is your environment" gating.

- 2026-06-05 — Inline IIFE pattern (script block in the page body, not deferred to app.js) is the correct architectural choice for any feature that needs to work BEFORE the full app loads. The min-screen-overlay overlay is the canonical example: users hitting it can't load the app, so the diagnostic UI has to be self-contained. Same pattern would apply to any "browser not supported", "JS disabled", "cookies blocked" gating in the future.

- 2026-06-05 — release-check validator caught no drift across v3.63.135 through v3.63.164 (thirty ceremonies).
