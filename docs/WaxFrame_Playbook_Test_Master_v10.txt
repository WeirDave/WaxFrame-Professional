──── SETUP 3 — YOUR PROJECT ────────────────────────────────
Project name:            Blog — Why I Stopped Trusting Single-Model AI for Network Docs
Project version:         v1.0
Document type:           Blog Post
Target audience:         Senior network engineers and IT leads who already use ChatGPT or Claude for technical documentation and have hit the wall where the output is "fine but not right." Reading on LinkedIn, Reddit r/networking, or a personal blog.
Desired outcome:         Reader finishes the post with a concrete reason to try running the same document through multiple models in sequence. They click through to the linked tool at the end.
Scope & constraints:     Cover: the one-shot trap (single model leaves blind spots), why different models catch different issues, a concrete network-engineering example with real numbers, and what the reader can try this week. Do not invent statistics. No marketing fluff.
Tone & voice:            First-person, direct, slightly sarcastic. Written the way I would say it to another network engineer over a beer, not the way I would write a vendor whitepaper. Voice is someone who's been burned by AI-generated config drift and finally found a workflow that doesn't suck.
Length:                  Target · 800 words
Additional instructions: No buzzwords. No "synergy" "leverage" "revolutionize" or any LinkedIn-influencer language. If a paragraph reads like a product brochure, cut it. Keep my voice — do not sand it down into something that could have been written by anyone.

──── SETUP 4 — REFERENCE MATERIAL ──────────────────────────
Thesis: any single LLM, no matter how good, has blind spots.
Run the same document through 4 or 5 different models in
sequence and the output is meaningfully better than any one
model can produce on its own.

Concrete example to reference:
- Took an Aruba ArubaOS 8.x configuration document, ~900
  words, describing an ArubaOS firmware-upgrade procedure
  for a multi-controller deployment
- Ran it through Claude alone: got a tightened version,
  maybe 15% better. Still missed two ordering issues that
  would have caused a controller failover during the upgrade
- Ran the same document through Claude → ChatGPT → Gemini →
  back to Claude: got a version where both ordering issues
  were caught, the prerequisites section was reorganized,
  and one assumption about RADIUS server reachability was
  flagged as needing verification before the change window

What the reader should try today: pick a runbook or
configuration document they care about, hand it to 3
different models in sequence with the same prompt, compare
the result against running it through any one model once.

Call to action: link to a tool that automates this — multiple
models, single document, convergence loop. (This is WaxFrame
but the post should not name it overtly — let the link
speak for itself.)
