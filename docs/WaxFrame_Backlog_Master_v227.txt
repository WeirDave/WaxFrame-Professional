2026-06-04

State: v3.63.158 Pro shipped on main. Twelfth Worker Bees release of the evening. One focused regression fix: sidebar "Jump to AI" links no longer just expand the row, they actually scroll the .fs-body-single container to bring the row into view.

Root cause was a wrong scroll-target: jumpToAISetupRow used window.scrollTo, but .fullscreen-layout has overflow:hidden so the window itself doesn't scroll on screen-bees. .fs-body-single is the actual overflow:auto container. The call was a no-op.

Fix:
- Replaced manual scrollTo math with row.scrollIntoView({behavior:'smooth', block:'start'}) — auto-finds the right scrollable ancestor.
- Wrapped in requestAnimationFrame so the freshly-rendered row is laid out before the scroll request.
- Added scroll-margin-top:80px to .ai-setup-row so the row lands 80px below the scrollport top (replaces hand-computed offset math).
- Added window.jumpToAISetupRow = jumpToAISetupRow binding (same pattern from v3.63.156/.157).

David's report: "the links on the left in the menu used to take you to that provider's card now it just seems to open up the card... if you have small screen and you click on it then you can jump all the way down without having to scroll". Closed the regression.

This was a regression introduced when the sidebar shipped in v3.63.153. I had used window.scrollTo without verifying which container was scrollable. Worth a release-check rule: any scrollTo / scrollIntoView call inside an event handler should be verified against the actual scrollable container of the target element.
