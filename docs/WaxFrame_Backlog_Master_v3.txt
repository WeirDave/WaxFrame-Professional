[ ] ★ #8 Length mode overhaul                 → v3.33.0
    Replace single lengthLimit field with explicit mode
    picker on Setup 3:
      • No limit    — silent, no gates
      • Hard cap    — ceiling only, no floor, no
                      undersized modal
      • Target      — strict at N, both floor and ceiling
                      at the target value, symmetric
                      guards armed
      • Range       — between min and max, both fields
                      user-supplied
    Builder prompt envelope must change per mode:
      • Hard cap: "Stay at or below N. Shorter is fine."
      • Target:   "Hit N exactly. Do not deliver less,
                   do not deliver more."
      • Range:    "Stay between min and max. Aim for the
                   middle."
    Migration: pre-v3.32.29 projects default to Hard cap
    mode with their existing lengthLimit copied across.
    Convergence-under and round-undersized intercepts
    only fire in Target or Range modes. Kills the implicit
    LENGTH_FLOOR_RATIO = 0.5 from v3.32.28 — replaced by
    explicit user-opted modes.
    SCOPE: data-model change. Touches Setup 3 UI, prompt
    envelope, all four length-guard intercepts, migration,
    helper-page docs. Probably v3.33.0 since it's a
    model change.

[ ] ★ Run mode toggle — Auto vs Interactive   → DESIGN FIRST
    NEW from session 2026-05-07. Two modes:
      • Auto        — runs continuously to convergence,
                      no halt at user-decision blocks
                      (unattended use case)
      • Interactive — current behavior; halts at user
                      decisions
    Auto mode REQUIRES guardrails. The 37-round Blog v2.0
    session would have run forever without one. Must define:
      • Max-rounds ceiling (default value? user-overridable?)
      • Stall detector ("3 consecutive rounds with no change
        in satisfied-count → halt and ask")
      • UI placement of the toggle (Setup 3? work toolbar?)
      • Halt-and-resume modal copy
    DELIVERY: design markdown spec first, then code in a
    separate release. Don't squeeze in with other work —
    architectural lever, deserves its own session.

[ ] #1 Fluid scaling architectural pass
    Footer scaling at 1872×938 was the trigger. Symptom
    of a deeper viewport-strategy problem: fixed px vs
    clamp/fluid units, the min-width: 1024px floor, the
    @media (min-width: 1601px) desktop-cascade pattern

    PRECONDITION: dedicated planning chat to produce

[ ] ★ Project-setup specificity drives convergence
    NEW from session 2026-05-07. Empirical finding:
    convergence speed correlates with project-setup
    specificity, not just document state. Same hive,
    same topic, v1.0 (specific goalNotes 243 chars,
    anchored persona, "Social Media Post" docType,
    800-word limit, Cl37S builder) converged in 5 rounds.
    v2.0 (generic goalNotes 48 chars, generic tone,
    "Blog Post" docType, 1500-word limit, Gem25P builder)
    ran 37 rounds without converging.
    Multiple correlated variables but the goalNotes
    shrinkage (243 → 48 chars) and docType flip
    (Social → Blog) are the two biggest contributors.
    LESSON FOR USERS: invest more time in setup
    specificity — detailed goalNotes and accurate
    docType pay off in fewer rounds.
    Connects to existing convergence-principle callout
    and model-diversity guidance below — they're all
    facets of the same broader theme: explicit
    constraints reduce convergence time.
    PLACEMENT TBD: user manual? document-playbooks.html?
    Both?

[ ] Builder selection heuristic
    NEW from session 2026-05-07. Companion finding to
    the project-setup specificity item above. Empirical
    pattern: same hive, Cl37S as Builder converged
    cleanly on multiple docs (Blog v1.0, LinkedIn post),
    Gem25P as Builder failed to converge on equivalent
    prose. Likely cause: Gem25P's Builder output invites
    more reviewer dissent (specifically from itself as
    a reviewer of its own output, plus from Cl46O).
    Worth documenting as a "Builder selection" guide
    in the user manual once we have more data points
    across doc types.

[ ] Convergence-principle callout
    "Starting from scratch converges faster than refining
    a heavily misaligned draft" — backed by the cook's-
    résumé-adapted-for-network-engineer example data
    point. Now ALSO backed by the v1.0 vs v2.0 setup-
    specificity finding above. Placement TBD: README,
    user manual, or playbooks. Open questions on
    placement priority and tone — not yet answered.

[ ] User-manual guidance on model diversity
    For model-server deployments: the Recommend-a-Model
    convergence trap. Minimum-two-distinct-models rule.
    Document somewhere in the user manual.
