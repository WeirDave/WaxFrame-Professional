WaxFrame Storage Schema v1
2026-06-12

ABOUT THIS DOCUMENT
────────────────────────────────────────────────────────────
First-pass inventory of every localStorage key written or
read by WaxFrame Professional. Produced for backlog item #7
(localStorage migration audit). Built from an AST scan of
every js/*.js file plus every inline <script> block in the
HTML helper pages — 224 setItem/getItem/removeItem
operations across 4 JS files and 2 HTML files, normalized
to 42 unique keys (29 static + 13 dynamic-name patterns).

The static enumeration found ZERO orphan keys. Two patterns
look orphan to a static reference scan but are intentional:

  • waxframe_v2_hive_recovery — write-only safety net.
    When loadSettings() can't parse LS_HIVE, the raw blob
    gets stashed here for the user to recover via DevTools.
    Read by humans, not by code. See storage.js:874.

  • waxframe_v2_settings — read-once, removed-once. The
    pre-v3.45 single-blob settings key, kept around so a
    long-cold browser profile from before that release
    migrates cleanly on first load. See storage.js:839-855.

What this document does NOT yet do (left as remaining work
on backlog #7):
  • Manual test plan for forwarding a v3.21-era profile
    through every release boundary. Needs runtime testing
    in a controlled browser; static enumeration is silent
    on whether each read path actually tolerates every
    historical shape it might encounter.

Audit script: .claude/scratch/ls-audit.js (gitignored).
Run with `node .claude/scratch/ls-audit.js <project-root>`
after installing acorn + acorn-walk into .claude/scratch/.

FORWARD-COMPAT TEST PLAN — v3.21 PROFILE THROUGH CURRENT
════════════════════════════════════════════════════════════
Manual test plan for confirming a long-running user's
localStorage survives every release boundary between v3.21
(the oldest WaxFrame Professional version still in the wild)
and current. Run this whenever the storage envelope changes
(envelope.version bump in storage.js) or whenever a per-key
shape changes in app.js. Each step describes the starting
state, the migration path expected to fire, and the post-
migration assertion you verify in DevTools.

A. SETUP — capture a v3.21 starting snapshot
   1. Open a fresh browser profile / clean Incognito window.
   2. Load WaxFrame at the v3.21 tag in localhost (git
      checkout v3.21, open file://… or python -m http.server).
      Note: vendor libs may have moved; use the v3.21 lib/
      directory not current.
   3. Configure: one default AI (Claude) with a real key,
      one custom AI (Open WebUI on localhost), one project
      with two saved checkpoints, one suppress-prompt
      dismissed (e.g. theme-help confirm).
   4. Export localStorage to a JSON file via
      `JSON.stringify(localStorage)` in the console. This
      is the FROZEN SNAPSHOT used for every following step.

B. MIGRATIONS to walk (in order)
   For each release range below, hot-swap to the target
   tag and load the page. Confirm: no thrown errors in
   console; the configured AI + project + checkpoints are
   visible; the suppress-prompt stays suppressed.

   v3.21 → v3.32  Envelope was pre-v6 in this range.
                  Storage.js read path should auto-migrate
                  envelope.version to v4. Key migrations
                  worth eyeballing: waxframe_v2_hive shape
                  change (api configs moved into hive),
                  waxframe_v2_settings retiring (read-only
                  legacy after v3.45).
   v3.32 → v3.45  envelope.version bump to v5. The pre-
                  v3.45 waxframe_v2_settings key gets
                  one-shot-migrated into the hive envelope
                  on first read. Confirm settings persisted
                  (theme, mute, devtoolbar position).
   v3.45 → v3.63  envelope.version bump to v6. Many per-
                  feature keys added across this range
                  (waxframe_dev, waxframe_hive_profile,
                  waxframe_conflict_ledger, the recommend
                  cache family, the models cache family,
                  the tiers cache, etc.). Each is
                  defaulted to empty/false at read time
                  if absent, so the v3.21 snapshot reaches
                  the work screen with all new toggles at
                  their defaults — no manual cleanup
                  required.
   v3.63 → current  Within v3.63.* the recommend cache
                  invalidates whenever waxframe_recommend_
                  promptver doesn't match the current
                  prompt-version sentinel (currently 'v2').
                  Confirm: load with `localStorage.
                  removeItem('waxframe_recommend_promptver')`
                  first; the IIFE at app.js L7657-7661
                  should clean the entire family on next
                  load.

C. PER-KEY SPOT CHECKS (after the full walk)
   For each key listed below, open DevTools and verify the
   expected post-migration shape:

   • waxframe_v2_hive       envelope { version: 6, payload: {...} }
   • waxframe_v2_settings   absent OR the legacy shape (we
                            keep it readable but stop
                            writing it)
   • waxframe_dev           'true' / 'false' (string)
   • waxframe_muted         'true' / 'false' (string)
   • waxframe_v2_theme      'light' / 'dark' / 'auto'
   • waxframe_hive_profile  { activeId, profiles: { [id]: {...} } }
   • waxframe_models_${p}   { ts: epoch, models: string[] } — for
                            each provider you keyed in step A.4
   • waxframe_recommend_default-${aiId}-reviewer
                            { ts, model, why, labels, none? } —
                            present iff Recommend ran for that
                            AI in this session
   • waxframe_v2_hive_recovery
                            absent (only set when the main
                            hive blob fails to parse — the

D. EXIT
   Tear down the test profile. Document any deviations
   from the expected post-migration shape and file as
   storage bugs in the backlog (or, if the deviation is

Cadence: run this test plan once per major envelope.version

════════════════════════════════════════════════════════════
SCHEMA VERSIONING CONVENTION (v3.63.302+)
════════════════════════════════════════════════════════════
Closes the last open item on backlog #6. Answers the
question "what shape am I looking at?" for every object-
payload localStorage entry in WaxFrame.

THE RULE
  • Object-shape payloads MAY carry a top-level
    `schema_version: <integer>` field.
  • Readers default to v1 when the field is absent. Every
    payload written before v3.63.302 is implicitly v1 — no
    in-the-wild localStorage needs to be touched.
  • Writers stamp the version via wfWriteVersioned() on
    every save (helper in storage.js).
  • Future shape changes bump the integer and supply a
    migrator chain `{ 2: v1=>v2, 3: v2=>v3, … }` to
    wfReadVersioned().
  • Newer-than-known payloads (downgrade case — a user
    upgraded WaxFrame on machine A and downgraded on
    machine B) are returned UNCHANGED with a console warn.
    Never stomped. The user keeps their data.

WHAT GETS IT
  Object payloads. Today every per-key cache that already
  stores a `{ ts, ... }` blob is a natural fit — minimal
  noise to add one field. Examples:
    • waxframe_models_${providerId}       ✓ adopted v3.63.302
    • waxframe_recommend_${cacheId}       ✓ adopted v3.63.302
    • waxframe_tiers_${providerId}        not yet — when next touched
    • waxframe_hive_profile               not yet — when next touched
    • waxframe_concurrency_overrides      not yet — flat object map

WHAT DOESN'T GET IT
  String / boolean / primitive payloads. The convention is
  for object shapes only. Adding `schema_version` to a
  1-character toggle like waxframe_v2_theme ('light' /
  'dark' / 'auto') would be silly. Examples that stay raw:
    waxframe_v2_theme, waxframe_muted, waxframe_dev,
    waxframe_decision_diff_mode, waxframe_recommend_promptver
    (a marker string, not a payload).

ADOPTION POLICY
  Per-key opt-in, not bulk-retrofit. 42 unique keys today;
  rewriting all of them at once would risk breaking
  something that doesn't need fixing. Instead: helpers live
  in storage.js, the convention is documented here, and
  each key adopts the helpers as its shape evolves OR as
  someone touches its read/write path for another reason.

THE HELPERS
  Both live in js/storage.js, exposed on `window` for
  cross-script use.

  wfReadVersioned(key, currentVersion, migrators?)
    Read a versioned payload. Walks the migrator chain to
    upgrade legacy payloads to currentVersion. Returns
    null on missing/invalid; returns the payload object
    on success.

  wfWriteVersioned(key, payload, currentVersion)
    Serialize and store an object payload with
    schema_version stamped. Quota/privacy-mode write
    failures return false; success returns true. Does
    NOT mutate the caller's payload (copies + stamps).

  Migrator shape:
    { 2: (payloadV1) => payloadV2,
      3: (payloadV2) => payloadV3,
      ... }
  Keys are the version being upgraded TO. Sparse maps are
  fine — a missing migrator step logs a warn and returns
  the payload at the highest version it could reach.

INTERACTION WITH CHECKPOINT-FILE FORMAT (v6 envelope)
  The checkpoint file format already carries its own
  `version` field at the outer envelope level — that's a
  different problem (export/import file shape) and is
  unaffected by this convention. The two coexist cleanly:
  per-key schema_version lives INSIDE the payload;
  checkpoint envelope version lives OUTSIDE it.

ROLLOUT STATUS
  v3.63.302  helpers shipped; convention documented;
             adopted on waxframe_models_${providerId} and
             waxframe_recommend_${cacheId}.
  Future    each remaining object-payload key adopts the
             helpers the next time its read/write path is
             touched — no scheduled bulk migration.
