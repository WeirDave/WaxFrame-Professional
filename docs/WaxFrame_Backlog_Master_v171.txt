Open Bugs

- importSession (Restore from Backup) writes LS_HIVE verbatim — does NOT normalize legacy IDs or stale display names. ROOT CAUSE: `storage.js:1416` does `localStorage.setItem(LS_HIVE, data.LS_HIVE)` for non-scrubbed imports, dumping the backup blob into localStorage with no transformation pass. Consequence: any backup file containing legacy ugly IDs (e.g. `cohere_1779690241686`, `together_ai_1779690170122`, `ai21__jamba__1780256643064` from pre-clean-id code, OR `cohere(2)`-style suffixes from the pre-v3.63.95 skeleton-collision bug) survives Restore intact, regardless of how many WaxFrame versions have shipped since. Also persists obsolete display names like "AI21 (Jamba)" from older QUICK_ADD_PROVIDERS presets. Confirmed reproducing 2026-05-31 on v3.63.95: David wiped localStorage, restored from a backup, and immediately had the same byte-identical timestamp-suffixed IDs back in his hive. Discovered while debugging David's "I wiped and re-added but they're still ugly" report — wipe + reload works correctly; Restore is the path bringing them back. WORKAROUND SHIPPED 2026-05-31: standalone tools/scrub-waxframe-backups.mjs script that processes backup files offline and rewrites them with clean IDs + clean display names, updating ALL cross-references (activeAIIds, keys map keys, models map keys, customAIs[].id+provider, customAIConfigs map keys + .label, builder, IDB_SESSION.sessionAIs, IDB_SESSION.ringBuffer[].ai/.provider). API key values are preserved byte-for-byte. Idempotent (clean files report no changes). Lives under tools/ (excluded from release ZIP per release-ceremony rule). PROPER IN-APP FIX: importSession should run a normalization pass on the imported hive before writing it to localStorage. Steps: (1) parse `data.LS_HIVE`; (2) for each customAIs[] entry, look up its endpoint in QUICK_ADD_PROVIDERS preset table and compute the clean id+name from the preset; (3) build oldId→newId rename map; (4) handle collisions with (N) suffix; (5) apply rename to all cross-references inside the parsed hive (same set the standalone script handles); (6) re-stringify and write. Same logic shape as the standalone script — could be ported into storage.js as a `normalizeHive()` helper called from importSession. Question to resolve at implementation time: should normalization fire silently, or surface a "Cleaned up N legacy IDs from this backup" toast so the user knows it happened. Lean toward the toast for transparency.

Open Features

- Selective backup EXPORT (producer-side counterpart to the diff-and-pick import flow below). CURRENT: backupSession at storage.js:959 captures the full local state — LS_HIVE, LS_PROJECT, LS_LICENSE, LS_SESSION, IDB_SESSION — into one JSON file with no choice over scope. PROPOSED: backup-generation modal lets the user check/uncheck what goes in the file: ☑ Project (goal, reference material, length settings) / ☑ Session in progress / ☐ Hive composition / ☐ API keys / ☐ Builder selection / ☑ License (default ON, untick when sharing with a different user). KEY DESIGN POINT — License is user-choosable but defaults ON: license is per-user not per-install, so for self-backups (work→home, machine→machine) the user wants license to ride along. BUT for coworker sharing (David's collaboration use case: hand a project to a teammate who has their own license and continues the work), the user needs to be able to strip it. Defaulting ON keeps the common case one-click; the checkbox covers the share case. RETIRES THE SCRUBBED-BACKUP SPECIAL CASE: today's `_waxframe_backup_scrubbed: true` flag and its receiver-side merge branch in importSession (storage.js lines 1398-1417) exist specifically to do this one thing — strip license + keys when sharing setups with different users. Once selective export ships, that special case is redundant: the user just unticks License + Keys at export time, accomplishing the same outcome through the general mechanism. Plan to delete the `_waxframe_backup_scrubbed` code path as part of this feature's implementation. KEY USE CASE — portable projects across machines: David starts a project at work using Anduril's Alfredo (internal AI gateway, only reachable from the work network), mails the backup to himself at home, wants to continue the project at home using his personal hive (different AIs, different keys). Today this is broken — restoring the work backup at home replaces home's hive with the work hive including the unreachable Alfredo entry. With selective export, at work the user unticks Hive + Keys + Builder (License stays ticked — same user on both ends). Mailed to home, imported via the diff-and-pick UI, the home hive is untouched and the project lands on top; license matches local so no change. SECOND USE CASE — coworker handoff: David mails the same project to a teammate. This time he ALSO unticks License (and probably Keys — teammate has their own). Teammate imports, gets the project + David's hive composition (so the same AIs participate going forward), but uses their own license and their own keys. ARCHITECTURAL SHAPE: the omitted sections write as `null` in the backup file (mirrors how `_waxframe_backup_scrubbed` backups already null out LS_LICENSE), so the format stays one envelope and the consumer's diff-and-pick UI naturally renders "Not included — your existing X will be used" for nulled sections. RELATED OPEN PROBLEM — session-AI mapping when a portable session lands on a hive missing some of its referenced AIs: `IDB_SESSION.sessionAIs[]` and `IDB_SESSION.ringBuffer[].ai/.provider` may reference AI IDs that don't exist in the destination hive (e.g. Alfredo at work, missing at home). Three reasonable behaviors: (1) drop unknown IDs from sessionAIs and ringBuffer, session continues with whatever local AIs remain; (2) drop with explicit user-facing warning "3 AIs from the original session aren't in your local hive: alfredo, [...]. Their past contributions are preserved in the document but they won't participate going forward"; (3) remap — let the user pick a local AI to fill in for each missing one, future-only, past contributions stay attributed to the original. Option 2 is the cleanest default. Idea credit: David 2026-05-31, while building on the diff-and-pick conversation — work/home split is his real workflow.

- Restore-from-Backup diff-and-pick UI — turn the current blunt overwrite (storage.js importSession at line 1416) into a comparison + selective-import flow. CURRENT: user picks a backup file, gets a one-shot trust-warning modal, then the entire backup state replaces local state with no preview of what's about to change. PROPOSED: after the trust-warning, before any writes happen, show a comparison modal that diffs backup vs current local state across these categories: (1) Hive composition — which AIs are in backup but not local (checkbox: import?), which are in local but not backup (checkbox: keep?); (2) API keys — keys matching, keys in backup only (import?), keys differing between backup and local (pick which to keep, per-key); (3) Per-AI model selections — when backup has a different model for the same AI, surface the choice; (4) Builder — when backup's Builder differs from local's, surface the choice; (5) Project + session — show backup project name vs current, show backup session round-count vs current, let user choose: project only / project + session / neither; (6) License — choice surfaced when relevant: if backup contains no license (was unticked at export) → row not shown; if backup license matches local → "License: matches your current install" (info-only, no checkbox); if backup has license and local doesn't → "License: import from backup ☑" (default ON); if backup has DIFFERENT license than local → "Backup license belongs to a different user — keep your current ☑ / use backup's ☐" (default to keeping local; never silently overwrite a valid local license); (7) Legacy IDs — once the importSession-normalize fix ships, show the diff using POST-normalization IDs so the user previews the final state, not the raw backup IDs. User checks/unchecks what they actually want, hits Apply, and WaxFrame writes only the chosen pieces. HEADLINE USE CASE — portable projects across hives (David's work/home split): work backup contains Anduril's Alfredo as Builder, unreachable at home; user imports at home, diff-and-pick lets them tick "import project + session" but leave "hive composition" + "keys" + "builder" unticked — the project lands on top of the home hive without trying to add an unreachable AI; license auto-resolves to "matches your current" (same user on both ends). Pairs with selective-export above (the producer-side counterpart) to make the round-trip clean. OTHER SCENARIOS THIS SOLVES: "I was 4 rounds into a session and accidentally restored a backup that wiped it" — diff shows "will replace your 4-round session in progress" before committing; "I just want the keys from this backup, not the whole hive composition" — selective import handles this directly; "Backup was from when I had Perplexity but I removed it later" — diff shows "Perplexity will be added back to your hive" with explicit checkbox to skip; "Did I just bring back ugly IDs?" — diff shows the final post-normalization IDs before any writes. ARCHITECTURAL NOTE: storage.js already has ONE special-case merge path for `_waxframe_backup_scrubbed: true` backups (line 1398-1417) that does exactly this kind of selective merge — preserves receiver's keys/license while importing sender's hive composition. This feature generalizes that one-off special case into the always-on flow, eliminating the need for a separate scrubbed-backup code path. Idea credit: David 2026-05-31, while explaining the restore-ID problem to Candy. Pairs with: the importSession ID-normalization bug fix above (the diff would show normalized IDs as part of the preview); and with the selective-export entry above (round-trip portability).

- Mistral 429 proactive warning via X-RateLimit-Remaining header. Mistral's official docs note their API returns an `X-RateLimit-Remaining` response header. WaxFrame doesn't currently read it. Adding header capture to the Mistral provider path (and any other provider that exposes the same shape) would let the troubleshooting layer surface a soft warning BEFORE a 429 hits — e.g. a yellow chip on the Mistral row that reads "Mistral: ~X requests left this minute" when the remaining count drops below a threshold (e.g. < 3). The 429 itself stays catalogued the same way; this just adds a pre-warning surface. Pairs with the API console URL split (right destination for usage info matters here too). Likely needs a small per-provider hook in the response-handling path in `callAPI` (app.js ~11500) since the available rate-limit headers differ by provider — Anthropic exposes `anthropic-ratelimit-*` series, OpenAI exposes `x-ratelimit-*-requests` and `x-ratelimit-*-tokens`, etc.

- Hive parallel fan-out RPS-awareness. CURRENT: a reviewer round fires up to N concurrent calls (one per AI in hive), where N can be 10+. With paid Mistral on Tier 1 or 2 (low RPS), that parallel burst itself can trigger 429 even though the user has plenty of monthly token budget remaining. CONSIDER: an optional per-provider concurrency cap so a Mistral request gets queued one-at-a-time if the user's tier requires it. UI surface: a Settings toggle "Throttle Mistral to 1 RPS (paid tier compatibility)" — off by default, user enables it if they keep getting 429d. The implementation could be a tiny per-provider mutex/queue in `callAPI`. Workaround until shipped: user contacts Mistral support to raise tier (per official Mistral docs that's their stated path to higher limits — there's no automatic spend-based promotion ladder).
