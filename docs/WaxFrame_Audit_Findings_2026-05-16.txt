FINDING 1 — Cross-page parity: setup screens missing theme +
            mute toggles  [CRITICAL]  — SHIPPED v3.40.1
═════════════════════════════════════════════════════════════

VERIFIED ROOT CAUSE: the 5 setup screens (Worker Bees,
Builder, Project, Reference, Document) use a `.fs-header`
pattern with three sections: left (hamburger + back), brand
(logo), and right (only the `.setup-step-badge`). No theme
toggle, no mute toggle anywhere in the header.

Compare against:
  • Work screen has them in `.work-topbar-right`
  • Helper pages have theme (not mute) in `.page-header-controls`
  • Welcome screen has theme in body (intentional placement,
    out of scope per David)

User-visible symptom: cannot change theme or toggle mute
while on setup screens. Forced to navigate to welcome or
work just to change theme.

FIX SHIPPED v3.40.1:
  • Wrapped each `.setup-step-badge` in a new
    `.fs-header-right` div containing the badge plus a mute
    button plus 3 theme buttons (light/auto/dark). 5 setup
    screens get identical markup.
  • Added `.fs-header-right` CSS rule mirroring
    `.work-topbar-right` layout values (flex/gap/alignment).
  • `_updateMuteBtn` widened from `#workMuteBtn` id-target
    to `.mute-btn` class-target so all mute buttons across
    the page reflect the same state.

──────────────────────────────────────────────────────────
FINDING 2 — Helper pages missing mute toggle  [ANNOYING]
            — SHIPPED v3.40.1
──────────────────────────────────────────────────────────

Helper pages (User Manual, Doc Playbooks, What Are Tokens,
API Details, Prompt Editor) have theme buttons in
`.page-header-controls` but no mute. Less critical than

FIX SHIPPED v3.40.1:

  • Extended `theme.js` with mute state machinery
    (`_isMuted`, `toggleMute`, class-targeting
    `_updateMuteBtn`). Helper pages don't load app.js, so
    theme.js mirrors the app.js definitions — same
    localStorage key (`waxframe_muted`) so state persists
    across page navigations.
  • Pattern matches the existing license-helper.js model
    ("self-contained mirror of in-app functions for
    helper pages").

──────────────────────────────────────────────────────────
FINDING 3 — Cache-bust drift across helper scripts  [COSMETIC]
            — SHIPPED v3.40.1
──────────────────────────────────────────────────────────

VERIFIED ROOT CAUSE: helper script cache-bust query strings
across the 6 HTML files were inconsistent before v3.40.1:
  • `js/theme.js` and `js/api-links.js`: NO `?v=` at all
    (cache-bust completely absent — any future broken
    version would never invalidate user browser caches)
  • `js/nav-helper.js?v=3.22.6` (frozen at old version)
  • `js/license-helper.js?v=3.22.6` (frozen at old version)
  • `js/docs-scrollspy.js?v=3.22.6` (frozen at old version)
  • `js/templates.js?v=3.38.11` (frozen at old version)

Latent maintenance issue, not directly causing any current
user-visible bug. But it's a landmine — when one of those
files gets edited, the cache-bust will need a bump that's
easy to forget.

FIX SHIPPED v3.40.1:
  • Standardized: every `<script src>` in every HTML file
    now carries `?v=3.40.1`. Discipline is now mechanical
    — every release bumps every cache-bust, no exceptions.
  • Theme.js and api-links.js cache-busts added
    (previously absent entirely).
  • Comment-header build stamps and meta build tags also
    swept to `20260516-002` across all 6 HTML files.

──────────────────────────────────────────────────────────
FINDING 6 — Legacy native confirm() calls in app.js
            [CRITICAL] — QUEUED for v3.41.1
──────────────────────────────────────────────────────────

Four spots in app.js still use the browser-native confirm()
instead of WaxFrame's wfConfirm() Promise-based modal:

  • Line 2818: license removal — "Remove your WaxFrame Pro
    license key from this browser?..."
  • Line 2903: home navigation — "Go back to the Home
    screen? Your session and document are saved..."
  • Line 8041: (truncated in grep, content unread —
    investigate during fix)
  • Line 8107: reference material mid-session —
    "Adding a new reference document mid-session takes
    effect on the NEXT round..."

The line 5978 occurrence is INTENTIONAL — it's the fallback
inside wfConfirm() itself for when the modal markup is
unavailable. Verified by reading the function body.

Why CRITICAL: native browser confirm() bypasses WaxFrame
styling, doesn't match the design language anywhere else in
the app, and varies in appearance across browsers. The
memory rule explicitly captures: "Browser native confirm()
dialogs replaced with Promise-based wfConfirm() modal styled
to WaxFrame design language." These four are leftover
migrations.

Fix: rewrite each call to use wfConfirm() with appropriate
title/message/buttons. ~4-line edit per call, plus async/
await context if not already in an async function.

──────────────────────────────────────────────────────────
FINDING 7 — Stray console.log in applyTemplate  [COSMETIC]
            — QUEUED
──────────────────────────────────────────────────────────

app.js line 4265:

    template card(s); ...`);

In-app diagnostics should use consoleLog() (which routes
to the WaxFrame console-overlay output) rather than the
raw browser console.log. Either migrate to consoleLog or

──────────────────────────────────────────────────────────
FINDING 8 — Toast wording inconsistency  [COSMETIC]
            — QUEUED
──────────────────────────────────────────────────────────

Toast messages across app.js vary in formatting:

  • Some use emoji prefix, some don't:
    "🐢 Slow-AI alerts: on" vs "Buffer is already empty"
  • Some end with period, some don't

═════════════════════════════════════════════════════════════
ADDENDUM 3 — POSTMORTEM: ZIP PACKAGING SILENT FAILURE
                        (v3.40.0 through v3.42.0)
═════════════════════════════════════════════════════════════

INCIDENT SUMMARY
─────────────────
v3.42.0 deployed and the live site immediately threw
`goToScreen is not defined` and `openNavMenu is not defined`
on click handlers. Investigation revealed the deployed
`js/app.js` was the OLD v3.40.0 build (`20260513-004`),
unchanged since that release.

ROOT CAUSE
──────────
ZIP packaging error in the release artifacts going back to
v3.40.0. The staging command:

  cp app.js  /tmp/zipstage/

placed `app.js` at the ROOT of the ZIP instead of at
`/tmp/zipstage/js/`. When extracted onto the repo folder,
the new code landed at `repo-root/app.js` — a location
nothing in index.html references. The actual
`repo-root/js/app.js` was never overwritten.

WHY IT WENT UNDETECTED FROM v3.40.0 THROUGH v3.41.0
────────────────────────────────────────────────────
v3.40.0 was the LAST release where `js/app.js` was correctly
deployed (presumably extracted manually or pushed before the
ZIP staging script was introduced).

v3.40.1, v3.41.0, and v3.42.0 all shipped with the broken
ZIP structure, but only v3.42.0 broke the live site. Reasons:

  • v3.40.1 (toggle parity, cache-bust sweep): pure cosmetic
    code; new HTML toggles didn't depend on any specific
    app.js state, and the OLD app.js's _updateMuteBtn just
    found no elements with .mute-btn class (no error).

  • v3.41.0 (theme/mute consolidation, audio extraction):
    NEW theme.js declared window._isMuted (global property),
    OLD app.js declared `let _isMuted` (script-level let).
    DIFFERENT BINDINGS — no collision. Audio play* functions
    redefined by NEW audio.js, OLD app.js redefined them
    again on its turn — function declarations don't error on
    conflict, just overwrite. Lucky coexistence.

  • v3.42.0 (scenes extraction): NEW scenes.js declares
    `let hiveFinishTimer`, `let _unanimousTimers`,
    `let _unanimousKeyHandler` at top-level script scope.
    OLD app.js declares the SAME identifiers as `let` at
    top-level script scope. Per spec, classic scripts share
    a global lexical environment; duplicate `let` is a
    SyntaxError. Browser stopped parsing app.js at the
    duplicate, never reached `goToScreen` definition.

DETECTION & FIX
───────────────
Detected via hash comparison between deployed app.js and
local app.js. Hashes differed; deployed file's BUILD stamp
revealed it was v3.40.0 vintage. Confirmed by running both
files through Node's V8 (V8 powers both Node and Chrome) —
both parse cleanly in isolation, but the runtime collision
between OLD app.js and NEW scenes.js happens only in
browser context where both scripts share the global scope.

Fix shipped as v3.42.1 (identical code, corrected ZIP
structure, cache-busts bumped to ?v=3.42.1 to force
fresh download).

PREVENTIVE MEASURE
──────────────────
New audit category added: Category 7 — Release Artifact
Verification. Required pre-delivery checks:

  unzip -l <zip> | grep -E "(^|/)app\.js$"
  → must show js/app.js only, never bare app.js at root

Plus three companion checks (no bare .js at root, directory
layout matches repo, file count sanity). See
WaxFrame_Audit_Methodology_v1.txt Category 7 for details.

LESSON
──────
The four canonical version stamps + cache-bust discipline +
syntax checks all PASSED for v3.40.1, v3.41.0, v3.42.0.
None of those checks verify the BOUNDARY between
"code is correct" and "code reaches production." That
boundary check was missing. It's now Category 7.

The deeper lesson — clean release ceremony is not the same
as clean release. The proof is in the deployment, not the
pre-flight.
