Observations

- P0 security cleared in one surgical release (v3.63.7). Both fixes followed the codebase's established two-layer trust-boundary pattern (import sanitization + render-sink belt for XSS; the redaction function + bundle-assembly redaction for content). safeRefId mirrors safeUrl exactly.

- safeRefId design: STRIP to [A-Za-z0-9_-] rather than regenerate, so legit ids pass through unchanged and the inline-handler lookups (which match by doc.id ===) stay intact. Only empty/all-bad ids fall back to a fresh generated id. Behaviorally tested: legit preserved, injection payloads stripped of all quotes/parens/brackets, empty falls back.

- The diagnostic redaction was leaking 3 ways despite the checkbox: LS_PROJECT raw (biggest — full document), plus ringBuffer + lastFailure inside the session that the redactor never visited. Lesson: when a redaction function exists, audit EVERY field that ends up in the export, not just the obvious docText/history — side-channels (debug buffers, separate localStorage blobs) are where content hides. consoleHTML included for completeness.

- Render-sink belt mutates doc.id in place when it's unsafe, persisting the normalized id so the rendered DOM id and the array id stay in sync (handlers look up by doc.id). Compute-once (single safeRefId call) avoids the regenerate-returns-different-value-each-call trap.
