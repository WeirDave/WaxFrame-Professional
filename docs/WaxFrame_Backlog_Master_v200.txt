Open Bugs

- importSession (Restore from Checkpoint) writes LS_HIVE verbatim — does NOT normalize legacy IDs or stale display names. ROOT CAUSE: storage.js setItem(LS_HIVE, data.LS_HIVE) for the v4 default path and the v5 hive-merge path both write the imported hive without an ID-normalization pass. Consequence: any checkpoint file containing legacy ugly IDs (e.g. `cohere_1779690241686`, `ai21__jamba__1780256643064` from pre-clean-id code, OR `cohere(2)`-style suffixes from the pre-v3.63.95 skeleton-collision bug) survives Restore intact. Confirmed reproducing 2026-05-31 on v3.63.95. PRIORITY NOTE 2026-06-04: zero confirmed downloads / zero sales means no user-in-the-wild has a checkpoint file that could carry these legacy IDs. The bug is real but its exposure surface is currently zero. Defer until either (a) the diff-and-pick UI work pulls in normalizeHive() as a side effect, or (b) actual users surface and one reports ugly IDs after a restore. PROPER IN-APP FIX: importSession should run a normalization pass on the imported hive before writing it to localStorage. REFERENCE IMPLEMENTATION: cleaner.html (offline tool, v1.2 build 20260601-skeleton-collision-fix) is a working version of normalizeHive() logic — validated against 68 real backup files, 64/68 cleaned correctly, 0 collision-suffix damage. Port the JS functions `deriveCleanFromUrl`, `deriveCleanFromId`, `isUgly`, `idTaken`, `normalizeBackup` from cleaner.html into a storage.js helper. CRITICAL IMPLEMENTATION NOTE FROM CLEANER v1.2 BUGFIX: the `idTaken` collision check MUST exclude the `models` map. Real claims live in `activeAIIds`, `keys`, `customAIs`, and `customAIConfigs` only. SECOND NOTE: 4th ID pattern from older Anduril/Alfredo work-gateway checkpoints (`_base__<provider>_<model_variant>_<13-digit-ts>_<index>`) passes the `__`-based isUgly check but no clean rename target exists — surface unrecognized-pattern ugly IDs to the user via a "Leave as-is or pick a clean name" prompt rather than guessing.
