Observations

- Architectural risk noted by Codex: API keys + license live in browser storage by design (acceptable for a local-first static app), which means ANY XSS becomes key theft. That's why the reference-doc import XSS (Next Up) is the highest-severity of the three security items — it turns a malicious backup into stored script execution against stored secrets.

- Restore-from-backup is now non-dismissable (v3.63.6). General principle reaffirmed: a trust gate you can permanently dismiss is not a trust gate. suppressKey belongs only on informational confirms. The restore confirm's own code comment already called it a social-engineering vector — the suppressKey contradicted that.

- Security-fix discipline this session: shipped only the two trivially-safe one-liners tonight (isEvalSupported:false + suppressKey removal). Deferred the two implementation-heavy trust-boundary fixes (redaction extension, XSS sanitization) to a dedicated session with the pre-surgery checkpoint and real testing. A botched security fix is worse than a known gap because it creates false confidence.

- isEvalSupported:false has zero functional cost for WaxFrame specifically because the app only extracts text from PDFs — it never renders interactive PDFs. The mitigation note in codeanlabs ("may impact PDFs relying on JS for interactive features") doesn't apply to our use case.

- Live-state audit (this session) confirmed all four earlier releases (v3.63.2-v3.63.5) are correctly pushed to GitHub at build 20260527-022. The "did we miss something" feeling traced to the pending security sweep, not a dropped change.

- docs/-in-zip policy refined: ship only the current backlog + rules reference. David keeps the 3-version rollback margin locally and auto-deletes older. The zip is not responsible for rollback history. (Earlier behavior shipped 3 backlog versions in the zip — corrected this session.)
