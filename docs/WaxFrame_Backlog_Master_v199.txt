Open Bugs

- importSession (Restore from Checkpoint) writes LS_HIVE verbatim — does NOT normalize legacy IDs or stale display names. ROOT CAUSE: `storage.js` setItem(LS_HIVE, data.LS_HIVE) for the v4 default path and the v5 hive-merge path both write the imported hive without an ID-normalization pass. Consequence: any checkpoint file containing legacy ugly IDs (e.g. `cohere_1779690241686`, `ai21__jamba__1780256643064` from pre-clean-id code, OR `cohere(2)`-style suffixes from the pre-v3.63.95 skeleton-collision bug) survives Restore intact. Confirmed reproducing 2026-05-31 on v3.63.95. PRIORITY NOTE 2026-06-04: David flagged that with zero confirmed downloads / zero sales to date, no user-in-the-wild has a checkpoint file that could carry these legacy IDs. The bug is real but its exposure surface is currently zero. Defer until either (a) the diff-and-pick UI work pulls in normalizeHive() as a side effect, or (b) actual users surface and one of them reports ugly IDs after a restore. PROPER IN-APP FIX: importSession should run a normalization pass on the imported hive before writing it to localStorage. REFERENCE IMPLEMENTATION: `cleaner.html` (offline tool, v1.2 build 20260601-skeleton-collision-fix) is a working version of `normalizeHive()` logic — validated against 68 real backup files, 64/68 cleaned correctly, 0 collision-suffix damage. Port the JS functions `deriveCleanFromUrl`, `deriveCleanFromId`, `isUgly`, `idTaken`, `normalizeBackup` from cleaner.html into a storage.js helper. CRITICAL IMPLEMENTATION NOTE FROM CLEANER v1.2 BUGFIX: the `idTaken` collision check MUST exclude the `models` map — that map carries skeleton-only entries from api.js's pre-registration, and counting them as real claims falsely triggers `(N)` suffix on legitimate renames. Real claims live in `activeAIIds`, `keys`, `customAIs`, and `customAIConfigs` only. SECOND NOTE: the cleaner's `deriveCleanFromId` recognizes three patterns (`prefix_<13-digit-ts>`, `prefix__name__<13-digit-ts>`, `name(N)`) but the 2026-06-02 audit surfaced a fourth pattern from older Anduril/Alfredo work-gateway checkpoints: `_base__<provider>_<model_variant>_<13-digit-ts>_<index>`. These pass the `__`-based isUgly check but the cleaner correctly skipped them (no URL match against the public-provider table, no clean rename target). For the in-app port: surface unrecognized-pattern ugly IDs to the user via a "Leave as-is or pick a clean name" prompt rather than guessing.

Open Features

- Restore-from-Checkpoint diff-and-pick UI — turn the current blunt overwrite into a comparison + selective-import flow. CURRENT (post-v3.63.130): selective EXPORT decides which sections leave the source machine; selective restore happens automatically via the v5 scope flags. But the receiver still has no preview of WHAT will change before commit — no way to see "this checkpoint will replace your 4-round session in progress" or "Perplexity will be added back to your hive" before clicking through. PROPOSED: after the trust-warning, before any writes happen, show a comparison modal that diffs checkpoint vs current local state across the same six sections the EXPORT modal exposes (Project / Session / Hive / Keys / Builder / License) plus per-AI key diff and per-AI model-selection diff. User checks/unchecks what they actually want from the file, hits Apply, and WaxFrame writes only the chosen pieces. Pairs naturally with the v5 scope semantics — the diff modal becomes "scope filter on RESTORE" the same way the save modal is "scope filter on SAVE". Scenarios this solves: "I was 4 rounds into a session and accidentally restored a checkpoint that wiped it" — diff shows "will replace your 4-round session" before committing; "I just want the keys from this checkpoint, not the whole hive composition" — selective restore handles this directly; "Did I just bring back ugly IDs?" — diff shows the final post-normalization IDs before any writes (pairs with the importSession normalizeHive() fix above). ARCHITECTURAL NOTE: this generalizes what the v5 EXPORT scope opt-in did to the SAVE side, applied symmetrically to the RESTORE side. STATUS 2026-06-04: PARKED — bigger scope, gated on user feedback once the selective-export modal actually has users.
