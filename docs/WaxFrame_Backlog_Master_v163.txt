Open Bugs

- "Send to Builder" button label/affordance unclear after "Edit document directly" on a single user decision. Reported 2026-05-31 mid-session. SCENARIO: round produces ONE user decision; user clicks "Edit document directly" in the decision modal; user saves their direct edit; the Send to Builder button at the bottom of the work screen stays lit. User is left wondering whether they need to click Send to Builder, or whether the round is now finished, or what action will close out the round. ACTUAL behavior: Send to Builder IS the right next click — "Edit document directly" applies the text edit and records the decision lock in `resolvedDecisions`, but the round itself doesn't close until the Builder-only round runs (which advances the round counter, clears the active conflict from the queue, and finalizes the document state for the next review round). PROBLEM: the button label "Send to Builder" doesn't communicate "you need to do this to finish the round" — especially after Edit Directly when the user already feels like they've "done the work." Several fixes worth considering: (1) re-label the button contextually — when a decision is locked-pending-finalization, show "Finalize Round" instead of "Send to Builder"; (2) add a subtle banner under the work-screen after Edit Directly: "Your edit is saved — click Send to Builder below to close the round"; (3) on Edit Directly save, briefly pulse/highlight the Send to Builder button so the user's eye is drawn to it; (4) consider auto-firing the Builder-only round on Edit Directly save when only one decision was pending, since the user has clearly resolved the only outstanding item — though this changes long-standing behavior and may surprise multi-decision users so probably not (1)/(2)/(3) are safer. Don't reword Send to Builder universally — it's the right label for the multi-decision case where AI picks were used. Context-specific label change is the cleanest fix.

Three viable approaches: (a) one-time hand-curated page with a "last updated YYYY-MM-DD" stamp, updated manually each release — lowest engineering cost, highest staleness risk; (b) a tools/ script that scrapes each provider's pricing/docs page nightly and emits a static `pricing.html` — moderate effort, fresh data, brittle on provider HTML changes; (c) per-provider pricing endpoint where available (very few providers expose this) plus manual fallback — most accurate, most maintenance. Recommend starting with (a) for immediate value, then graduate to (b) as a tools/ script that emits the same page format. Page columns to include: provider, model name, input $ per million tokens, output $ per million tokens, context window, max output tokens (matters for Builder-capability filtering — pairs with the Builder-capability entry below), per-call cost estimate for a typical WaxFrame review round (~5K prompt + 2K completion = ~7K tokens). Link from the existing api-details.html page so users hit it before picking a Builder or hive composition.

- Builder-capability filter at recommendation/selection time — flag models that can't fulfill the Builder role. Empirically confirmed broken Builders surfaced from production deep-dives so far: the entire `jamba-*` family (jamba-mini-2-2026-01 confirmed via 2026-05-31 BAGAIL session deep-dive with TWO distinct failure modes — outputs the refMaterial scaffold verbatim instead of synthesizing reviewer responses into polished prose, AND hits the 4096 max_tokens cap before reaching the %%CONFLICTS_START%% block; jamba-large-1.7-2025-07 separately confirmed by David in the same testing window — both Jamba models fail as Builder, so the whole family goes on the Builder-incapable list, Reviewer-only). Filter should run at: (1) Change Builder modal — greyed/warning-badged option with tooltip "May be unable to follow Builder formatting — try Reviewer role instead"; (2) Recommend Models pipeline — skip Builder-incapable models from Builder recommendations entirely. Mechanism shape: a `builderCapable: boolean` flag on the model metadata, populated from a known-bad list initially (start: all jamba-*), augmented over time by capture from ring-buffer evidence (finishReason === "length" on Builder calls + missing %%CONFLICTS_START%% pattern). PAIRED FIX — surface a clearer error than the generic BUILDER_NO_CONFLICTS_BLOCK when finishReason === "length": say something like "Builder output was truncated at the model's token cap — try a model with higher output capacity. This model hit max_tokens before completing the required formatting block." Same general shape as the existing "Recommend-time filter for instruction-incapable models (e.g. antigravity-preview)" parking-lot item — that's the same pattern at a different layer. UX cost of NOT shipping this: new user picks a cheap-but-incapable Builder, hits the failure, has no actionable diagnosis, gives up on the product. Real conversion risk.
