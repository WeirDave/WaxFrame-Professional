Open Bugs

- importSession (Restore from Checkpoint) writes LS_HIVE verbatim — does NOT normalize legacy IDs or stale display names. ROOT CAUSE: `storage.js:1416` does `localStorage.setItem(LS_HIVE, data.LS_HIVE)` for non-scrubbed imports, dumping the imported blob into localStorage with no transformation pass. Consequence: any checkpoint file containing legacy ugly IDs (e.g. `cohere_1779690241686`, `together_ai_1779690170122`, `ai21__jamba__1780256643064` from pre-clean-id code, OR `cohere(2)`-style suffixes from the pre-v3.63.95 skeleton-collision bug) survives Restore intact, regardless of how many WaxFrame versions have shipped since. Also persists obsolete display names like "AI21 (Jamba)" from older QUICK_ADD_PROVIDERS presets. Confirmed reproducing 2026-05-31 on v3.63.95: David wiped localStorage, restored from a checkpoint, and immediately had the same byte-identical timestamp-suffixed IDs back in his hive. Discovered while debugging David's "I wiped and re-added but they're still ugly" report — wipe + reload works correctly; Restore is the path bringing them back. NO STANDALONE WORKAROUND IN THE REPO. A `tools/scrub-waxframe-backups.mjs` script was discussed 2026-05-31 as an offline cleanup path, but it never landed in version control — `tools/` on `main` (verified 2026-06-01) contains only the screenshot-capture pipeline files. The standalone path is now considered obsolete: stacking the in-app `normalizeHive()` fix below with the selective-import diff-and-pick flow (under Open Features) cleans legacy checkpoints on the way in, without needing an offline tool. Each legacy checkpoint the user restores gets normalized + section-picked, producing a clean final state. PRIORITY NOTE 2026-06-04: David flagged that with zero confirmed downloads / zero sales to date, no user-in-the-wild has a checkpoint file that could carry these legacy IDs. The bug is real but its exposure surface is currently zero. Defer aggressive prioritization until either (a) the selective-export arc ships and lands `normalizeHive()` as a side effect, or (b) actual users surface and one of them reports ugly IDs after a restore. Keep entry as reference. PROPER IN-APP FIX: importSession should run a normalization pass on the imported hive before writing it to localStorage. Steps: (1) parse `data.LS_HIVE`; (2) for each customAIs[] entry, look up its endpoint in QUICK_ADD_PROVIDERS preset table and compute the clean id+name from the preset; (3) build oldId→newId rename map; (4) handle collisions with (N) suffix; (5) apply rename to all cross-references inside the parsed hive (same set the standalone script handles); (6) re-stringify and write. Same logic shape as the standalone script — could be ported into storage.js as a `normalizeHive()` helper called from importSession. Question to resolve at implementation time: should normalization fire silently, or surface a "Cleaned up N legacy IDs from this checkpoint" toast so the user knows it happened. Lean toward the toast for transparency. REFERENCE IMPLEMENTATION: `cleaner.html` (offline tool, v1.2 build 20260601-skeleton-collision-fix) is a working version of `normalizeHive()` logic — validated against 68 real backup files from David's pile, 64/68 cleaned correctly, 0 collision-suffix damage. Port the JS functions `deriveCleanFromUrl`, `deriveCleanFromId`, `isUgly`, `idTaken`, `normalizeBackup` from cleaner.html into a storage.js helper. CRITICAL IMPLEMENTATION NOTE FROM CLEANER v1.2 BUGFIX: the `idTaken` collision check MUST exclude the `models` map — that map carries skeleton-only entries from api.js's pre-registration (cohere, together, deepseek), and counting them as real claims falsely triggers `(N)` suffix on legitimate renames. Mirrors the v3.63.95 in-app fix in `makeCleanProviderId.taken()`. Real claims live in `activeAIIds`, `keys`, `customAIs`, and `customAIConfigs` only. SECOND IMPLEMENTATION NOTE: the cleaner's `deriveCleanFromId` recognizes three patterns (`prefix_<13-digit-ts>`, `prefix__name__<13-digit-ts>`, `name(N)`) — but the 2026-06-02 audit of David's 68-file pile surfaced a fourth pattern from older Anduril/Alfredo work-gateway checkpoints: `_base__<provider>_<model_variant>_<13-digit-ts>_<index>` (e.g. `_base__claude_3_7_sonnet_1776393640151_0`). These pass the `__`-based isUgly check but the cleaner correctly skipped them because (a) no URL match against the public-provider table, (b) no clean rename target — multiple Alfredo models behind one URL would collide if stripped to bare provider names. For the in-app port: surface unrecognized-pattern ugly IDs to the user via a "Leave as-is or pick a clean name" prompt rather than guessing. Don't invent rename rules for custom-gateway entries the cleaner can't resolve unambiguously.

Defaulting ON keeps the common case one-click; the checkbox covers the share case. RETIRES THE SCRUBBED-CHECKPOINT SPECIAL CASE: today's `_waxframe_backup_scrubbed: true` flag and its receiver-side merge branch in importSession (storage.js lines 1398-1417) exist specifically to do this one thing — strip license + keys when sharing setups with different users. Once selective export ships, that special case is redundant: the user just unticks License + Keys at export time, accomplishing the same outcome through the general mechanism. Plan to delete the `_waxframe_backup_scrubbed` code path as part of this feature's implementation. KEY USE CASE — portable projects across machines: David starts a project at work using Anduril's Alfredo (internal AI gateway, only reachable from the work network), mails the checkpoint to himself at home, wants to continue the project at home using his personal hive (different AIs, different keys). Today this is broken — restoring the work checkpoint at home replaces home's hive with the work hive including the unreachable Alfredo entry. With selective export, at work the user unticks Hive + Keys + Builder (License stays ticked — same user on both ends). Mailed to home, restored via the diff-and-pick UI, the home hive is untouched and the project lands on top; license matches local so no change. SECOND USE CASE — coworker handoff: David mails the same project to a teammate. This time he ALSO unticks License (and probably Keys — teammate has their own). Teammate restores, gets the project + David's hive composition (so the same AIs participate going forward), but uses their own license and their own keys. ARCHITECTURAL SHAPE: the omitted sections write as `null` in the checkpoint file (mirrors how `_waxframe_backup_scrubbed` checkpoints already null out LS_LICENSE), so the format stays one envelope and the consumer's diff-and-pick UI naturally renders "Not included — your existing X will be used" for nulled sections. Format bump v4 → v5: v5 carries `_waxframe_backup_scope: {project, session, hive, keys, builder, license}` so importSession can distinguish "partial intentionally omitted X" from "old all-or-nothing checkpoint with X = null". RELATED OPEN PROBLEM — session-AI mapping when a portable session lands on a hive missing some of its referenced AIs: `IDB_SESSION.sessionAIs[]` and `IDB_SESSION.ringBuffer[].ai/.provider` may reference AI IDs that don't exist in the destination hive (e.g. Alfredo at work, missing at home). Three reasonable behaviors: (1) drop unknown IDs from sessionAIs and ringBuffer, session continues with whatever local AIs remain; (2) drop with explicit user-facing warning "3 AIs from the original session aren't in your local hive: alfredo, [...]. Their past contributions are preserved in the document but they won't participate going forward"; (3) remap — let the user pick a local AI to fill in for each missing one, future-only, past contributions stay attributed to the original. Option 2 is the cleanest default.
