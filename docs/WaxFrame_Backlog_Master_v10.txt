[ ] ★ BUG: Multi-USER-DECISION + Lock-via-Notes ordering UX
    SURFACED: 2026-05-10 during T2 Executive Summary run.

    REPRO PATH:
      1. A round produces multiple USER DECISION conflicts
      2. User picks Option A on Decision 1 and clicks "Lock
         this in for next round" (or whatever the lock-via-
         Notes action is called)
      3. Notes drawer opens with the locked selection injected
      4. User sees the Send-to-Builder button highlighted
      5. Other USER DECISIONs (Decision 2, 3, ...) are STILL
         PENDING but no longer visible from the Notes drawer

    THE CONFUSION:
      • If user clicks Send to Builder NOW → does the round
        proceed with Decision 1 locked but Decisions 2/3
        unresolved? (they probably get the Builder's call,
        which silently overrides the user)
      • If user closes Notes drawer first → can they get back
        to Decisions 2/3 to lock those too? Or do they have
        to use the per-decision lock action again?

    SEVERITY: MEDIUM. Doesn't break runs but creates a
    "trust failure" — David doesn't know if his locked
    decisions will be honored when other unresolved decisions
    exist.

    PROPOSED FIX (design TBD, multiple options):

      Option A — MULTI-LOCK MODE:
        When multiple USER DECISIONs are pending and user
        clicks "Lock & open Notes", show a banner in the
        Notes drawer: "1 of 3 decisions locked — close Notes
        to lock the other 2." Send-to-Builder button gates
        on all decisions being resolved (locked OR resolved
        another way).

      Option B — LOCK-ALL FIRST, THEN NOTES:
        When user clicks lock on Decision 1, the action is
        "queue this lock — do not open Notes yet." After
        all decisions are locked, ONE Notes drawer opens
        with all locked text consolidated. User reviews,
        edits if needed, sends to Builder.

      Option C — INLINE LOCK + DETACH NOTES:
        Lock action immediately commits the selection without
        opening Notes (a checkmark or visual indicator next
        to the locked option). Notes drawer remains a
        separate, manual action. User locks all decisions
        first, then optionally opens Notes to add context.

    ACTION: defer until next session focuses on this. Worth
    making the right call up front rather than rushing.

[ ] ★ Engaged-time tracking for runs  (NEW v9)
    NEW from session 2026-05-10. Wall-clock session time is
    misleading because it includes idle periods when the user
    stepped away (lunch break, dinner, distracted by other
    work). The Publix potato salad recipe run shows "47 min"
    wall-clock; actual engaged time was a fraction of that.

    PROBLEM IT CAUSES: HISTORICAL RUNS times aren't comparable
    across runs. Two 4-round runs that both show "30 min" might
    actually have very different engaged times. This degrades
    the value of the time-component of measurement data.

    DESIGN OPTIONS:
      Option A (manual pause): add a Pause button next to the
      Auto/Manual control. User clicks before stepping away,
      clicks again on return. Simple, reliable, requires user
      discipline. The existing Auto-pause-on-Notes-open guard
      from v3.36.17 is a similar pattern — extend it.

      Option B (auto-detect idle): track last user-interaction
      timestamp (keystroke, click, scroll). If idle > N minutes
      (configurable, e.g., 5 min default), pause the round
      timer. Resume on next interaction. Smart but heuristic.

      Option C (both): manual pause for known breaks, auto-
      detect for forgotten ones. Best UX but most code.

    WHAT IT TOUCHES:
      • Round timer (currently wall-clock-based)
      • Console log timestamps (also wall-clock — these are
        good as-is for forensic purposes; engaged-time is a
        separate metric)
      • Transcript export "Session duration" line — should
        report both wall-clock AND engaged time for clarity
      • HISTORICAL RUNS rows in test master — going forward
        should record both metrics
      • Round badge / topbar elapsed display — show engaged
        time, not wall-clock

    NOT a blocker for current work — runs already in the wild
    have wall-clock data, that's what we have, we move on.
    Going forward, capturing engaged time would make the data
    more meaningful.

    PRIORITY: high but not P1 — affects measurement quality,
    not blocking any release. Lift when there's a quiet
    session with no other surgical work to do.
